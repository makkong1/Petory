# 4. 성능 최적화

## Q4-1. Board 도메인에서 301개 쿼리를 3개로 줄인 방법을 설명해주세요.

### 답변 포인트
- **문제**: 게시글 목록 조회 시 작성자, 반응 정보, 첨부파일을 개별 쿼리로 조회
- **해결**:
  1. Fetch Join으로 작성자 정보 함께 조회
  2. 배치 조회로 반응 정보 IN 절 집계
  3. 배치 조회로 첨부파일 정보 조회
- **결과**: 301개 → 3개 쿼리 (99% 감소)

### 상세 답변

#### 1. 문제 상황 분석
**위치**: `domain/board/service/BoardService.java` (최적화 전)

**Before 최적화 쿼리 수**:
```
1. 게시글 목록 조회: 1개 쿼리
2. 각 게시글의 작성자 조회: 100개 쿼리 (N+1 문제)
3. 각 게시글의 반응 정보 조회: 100개 쿼리
4. 각 게시글의 첨부파일 조회: 100개 쿼리

총 301개 쿼리
```

#### 2. 해결 방법

##### 2-1. Fetch Join으로 작성자 정보 조회
**위치**: `domain/board/repository/SpringDataJpaBoardRepository.java`
**메서드**: `findAllByIsDeletedFalseOrderByCreatedAtDesc()`

**전체 흐름**:
```
BoardService.getAllBoards()
  ↓
BoardRepository.findAllByIsDeletedFalseOrderByCreatedAtDesc()
  ↓
JOIN FETCH로 작성자 정보 함께 조회
  ↓
1개 쿼리로 게시글 + 작성자 정보 조회
```

**코드 예시**:
```java
// domain/board/repository/SpringDataJpaBoardRepository.java
@Query("SELECT b FROM Board b JOIN FETCH b.user u " +
       "WHERE b.isDeleted = false AND u.isDeleted = false AND u.status = 'ACTIVE' " +
       "ORDER BY b.createdAt DESC")
List<Board> findAllByIsDeletedFalseOrderByCreatedAtDesc();
```

**효과**: 100개 쿼리 → 1개 쿼리

##### 2-2. 배치 조회로 반응 정보 조회
**위치**: `domain/board/service/BoardService.java`
**메서드**: `mapBoardsWithReactionsBatch()`

**전체 흐름**:
```
게시글 목록 조회 완료
  ↓
게시글 ID 리스트 추출
  ↓
BoardReactionRepository.findByBoardIdxIn(boardIds)
  ↓
IN 절로 반응 정보 한 번에 조회
```

**코드 예시**:
```java
// domain/board/service/BoardService.java
private List<BoardDTO> mapBoardsWithReactionsBatch(List<Board> boards) {
    List<Long> boardIds = boards.stream()
        .map(Board::getIdx)
        .collect(Collectors.toList());
    
    // 배치 조회: 반응 정보
    Map<Long, List<BoardReaction>> reactionsMap = 
        boardReactionRepository.findByBoardIdxIn(boardIds)
            .stream()
            .collect(Collectors.groupingBy(BoardReaction::getBoardIdx));
    
    // 매핑
    return boards.stream()
        .map(board -> boardConverter.toDTO(board, reactionsMap, filesMap))
        .collect(Collectors.toList());
}
```

**효과**: 100개 쿼리 → 1개 쿼리

##### 2-3. 배치 조회로 첨부파일 정보 조회
**위치**: `domain/file/service/AttachmentFileService.java`

**전체 흐름**:
```
게시글 ID 리스트 추출
  ↓
AttachmentFileService.getFilesByTargetIds(boardIds, FileTargetType.BOARD)
  ↓
IN 절로 첨부파일 정보 한 번에 조회
```

**효과**: 100개 쿼리 → 1개 쿼리

#### 3. 최종 결과
**After 최적화 쿼리 수**:
```
1. 게시글 + 작성자 조회 (Fetch Join): 1개 쿼리
2. 반응 정보 배치 조회 (IN 절): 1개 쿼리
3. 첨부파일 배치 조회 (IN 절): 1개 쿼리

총 3개 쿼리
```

**성능 개선**:
- 쿼리 수: 301개 → 3개 (**99% 감소**)
- 실행 시간: 745ms → 30ms (**24.83배 개선**)

## Q4-2. 인기글 스냅샷 패턴을 설명해주세요.

### 답변 포인트
- 복잡한 계산 결과를 미리 생성하여 저장
- 스케줄러로 주기적으로 스냅샷 생성
- 조회 시 복잡한 계산 없이 스냅샷 조회
- 실시간성과 성능의 트레이드오프

### 상세 답변

#### 1. 스냅샷 패턴 개념
**위치**: `domain/board/service/BoardPopularityService.java`
**엔티티**: `domain/board/entity/BoardPopularitySnapshot.java`

**전체 흐름**:
```
스케줄러 실행 (매일 18:30, 매주 월요일 18:30)
  ↓
BoardPopularityService.generateSnapshots()
  ↓
인기글 계산 (조회수, 좋아요, 댓글 수 등)
  ↓
BoardPopularitySnapshot 엔티티로 저장
  ↓
조회 시 스냅샷 테이블에서 조회
```

#### 2. 스케줄러 설정
**위치**: `domain/board/service/BoardPopularityScheduler.java`

**코드 예시**:
```java
// 매일 오후 6시 30분에 주간 인기글 스냅샷 생성
@Scheduled(cron = "0 30 18 * * ?")
@Transactional
public void generateWeeklyPopularitySnapshots() {
    boardPopularityService.generateSnapshots(PopularityPeriodType.WEEKLY);
}

// 매주 월요일 오후 6시 30분에 월간 인기글 스냅샷 생성
@Scheduled(cron = "0 30 18 ? * MON")
@Transactional
public void generateMonthlyPopularitySnapshots() {
    boardPopularityService.generateSnapshots(PopularityPeriodType.MONTHLY);
}
```

#### 3. 스냅샷 생성 로직
**위치**: `domain/board/service/BoardPopularityService.java`
**메서드**: `generateSnapshots()`

**전체 흐름**:
```
기간 설정 (주간/월간)
  ↓
기간 내 게시글 조회
  ↓
인기 점수 계산 (조회수, 좋아요, 댓글 수 가중치 적용)
  ↓
상위 N개 선정
  ↓
BoardPopularitySnapshot 엔티티 생성 및 저장
```

**인기 점수 계산 공식**:
```
인기 점수 = (조회수 × 1) + (좋아요 × 2) + (댓글 수 × 3)
```

#### 4. 스냅샷 조회
**위치**: `domain/board/repository/BoardPopularitySnapshotRepository.java`

**코드 예시**:
```java
@Query("SELECT s FROM BoardPopularitySnapshot s " +
       "WHERE s.periodType = :periodType " +
       "ORDER BY s.popularityScore DESC")
List<BoardPopularitySnapshot> findByPeriodTypeOrderByPopularityScoreDesc(
    PopularityPeriodType periodType);
```

**효과**:
- 복잡한 계산 없이 단순 조회
- 조회 성능 대폭 향상
- 실시간성은 약간 떨어지지만 성능 우선

## Q4-4. Location 도메인에서 초기 로드 성능을 어떻게 개선했나요?

### 답변 포인트
- **문제**: 초기 로드 시 전체 데이터(22,699개) 조회
- **해결**: 사용자 위치 기반 10km 반경 검색
- **결과**: 22,699개 → 1,026개 (95.5% 감소)

### 상세 답변

#### 1. 문제 상황
**Before 최적화**:
- 초기 로드 시 전체 위치 서비스 조회: 22,699개
- 불필요한 데이터까지 조회
- 초기 로드 시간: 느림

#### 2. 해결 방법
**위치**: `domain/location/service/LocationServiceService.java`
**메서드**: `searchLocationServicesByLocation()`

**전체 흐름**:
```
사용자 위치 정보 수신 (위도, 경도)
  ↓
LocationServiceService.searchLocationServicesByLocation()
  ↓
LocationServiceRepository.findByRadius()
  ↓
ST_Distance_Sphere로 10km 반경 계산
  ↓
반경 내 위치 서비스만 조회
```

**코드 예시**:
```java
// domain/location/service/LocationServiceService.java
public List<LocationServiceDTO> searchLocationServicesByLocation(
        Double latitude, Double longitude,
        Integer radiusInMeters, String category, Integer maxResults) {
    
    // 반경 검색 수행
    List<LocationService> services = locationServiceRepository
            .findByRadius(latitude, longitude, (double) radiusInMeters);
    
    // 카테고리 필터링
    if (StringUtils.hasText(category)) {
        services = services.stream()
            .filter(service -> {
                // category3, category2, category1 순서로 확인
                // ...
            })
            .collect(Collectors.toList());
    }
    
    return services.stream()
        .map(locationServiceConverter::toDTO)
        .collect(Collectors.toList());
}
```

**ST_Distance_Sphere 쿼리**:
```java
// domain/location/repository/SpringDataJpaLocationServiceRepository.java
@Query(value = "SELECT * FROM locationservice WHERE " +
               "latitude IS NOT NULL AND longitude IS NOT NULL AND " +
               "ST_Distance_Sphere(POINT(longitude, latitude), POINT(?2, ?1)) <= ?3 AND " +
               "(is_deleted IS NULL OR is_deleted = 0) " +
               "ORDER BY rating DESC", 
       nativeQuery = true)
List<LocationService> findByRadius(Double latitude, Double longitude, Double radiusInMeters);
```

#### 3. 최종 결과
**After 최적화**:
- 사용자 위치 기준 10km 반경: 1,026개
- **95.5% 데이터 감소**
- 초기 로드 시간: 대폭 개선

## Q4-5. 배치 조회(Batch Query)를 어떻게 구현했나요?

### 답변 포인트
- IN 절을 활용한 집계 쿼리
- 500개 단위로 배치 처리하여 IN 절 크기 제한
- 예시: 반응 정보 배치 조회로 100개 쿼리 → 1개 쿼리

### 상세 답변

#### 1. 배치 조회 개념
**위치**: `domain/board/service/BoardService.java`
**메서드**: `mapBoardsWithReactionsBatch()`

**전체 흐름**:
```
게시글 목록 조회 완료
  ↓
게시글 ID 리스트 추출
  ↓
IN 절로 연관 데이터 배치 조회
  ↓
메모리에서 매핑
```

#### 2. 반응 정보 배치 조회
**위치**: `domain/board/repository/BoardReactionRepository.java`

**코드 예시**:
```java
// domain/board/service/BoardService.java
List<Long> boardIds = boards.stream()
    .map(Board::getIdx)
    .collect(Collectors.toList());

// 배치 조회: 반응 정보
Map<Long, List<BoardReaction>> reactionsMap = 
    boardReactionRepository.findByBoardIdxIn(boardIds)
        .stream()
        .collect(Collectors.groupingBy(BoardReaction::getBoardIdx));
```

**Repository 쿼리**:
```java
// domain/board/repository/SpringDataJpaBoardReactionRepository.java
@Query("SELECT br FROM BoardReaction br WHERE br.board.idx IN :boardIds")
List<BoardReaction> findByBoardIdxIn(@Param("boardIds") List<Long> boardIds);
```

**효과**: 100개 쿼리 → 1개 쿼리

#### 3. 배치 크기 제한
**주의사항**: IN 절 크기 제한

**해결 방법**:
- 500개 단위로 배치 처리
- 큰 리스트는 여러 번 나눠서 조회

**코드 예시**:
```java
// 500개 단위로 배치 처리
int batchSize = 500;
List<List<Long>> batches = new ArrayList<>();
for (int i = 0; i < boardIds.size(); i += batchSize) {
    batches.add(boardIds.subList(i, Math.min(i + batchSize, boardIds.size())));
}

Map<Long, List<BoardReaction>> reactionsMap = new HashMap<>();
for (List<Long> batch : batches) {
    boardReactionRepository.findByBoardIdxIn(batch)
        .forEach(reaction -> {
            reactionsMap.computeIfAbsent(reaction.getBoardIdx(), k -> new ArrayList<>())
                .add(reaction);
        });
}
```

#### 4. 첨부파일 배치 조회
**위치**: `domain/file/service/AttachmentFileService.java`

**코드 예시**:
```java
// domain/board/service/BoardService.java
Map<Long, List<FileDTO>> filesMap = 
    attachmentFileService.getFilesByTargetIds(boardIds, FileTargetType.BOARD);
```

**효과**: 100개 쿼리 → 1개 쿼리
