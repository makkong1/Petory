> **목적**: `01_프로젝트_개요_아키텍처.md` Q1-4를 보강하기 위한 상세 문서

## Part 1. 공통 개념·정의 정리

### 1.1 N+1 문제 (N+1 Query Problem)

| 항목 | 설명 |
|------|------|
| **정의** | 부모 엔티티 1개 조회 후, 자식/연관 엔티티를 N번 개별 조회하여 총 1+N개 쿼리가 발생하는 현상 |
| **원인** | JPA LAZY 로딩 시 연관 엔티티 접근마다 별도 쿼리 실행 |
| **영향** | 쿼리 수가 데이터 양에 비례해 선형 증가 → DB 부하, 응답 지연 |
| **발생 도메인** | Board, User(로그인), Care, Comment 등 |

---

### 1.2 Fetch Join

| 항목 | 설명 |
|------|------|
| **정의** | JPQL `JOIN FETCH`로 연관 엔티티를 한 번의 쿼리에 함께 조회하는 기법 |
| **특징** | LAZY 로딩 대신 즉시 로딩, 추가 쿼리 없음 |
| **적용 예** | `SELECT b FROM Board b JOIN FETCH b.user u` |
| **주의** | 컬렉션 Fetch Join 시 카테시안 곱(중복 행) 발생 가능 → `DISTINCT` 또는 별도 조회 필요 |

---

### 1.3 배치 조회 (Batch Query)

| 항목 | 설명 |
|------|------|
| **정의** | 여러 ID를 `IN` 절로 묶어 한 번에 조회한 뒤, 메모리에서 Map으로 매핑하는 패턴 |
| **흐름** | ① ID 목록 추출 → ② `WHERE id IN (:ids)` 쿼리 → ③ `Map<ID, Entity>`로 변환 → ④ 메모리 매핑 |
| **장점** | N번 쿼리 → 1~2번 쿼리로 감소, 게시글/채팅방 수와 무관하게 쿼리 수 일정 |
| **적용 도메인** | Board(반응·첨부파일), User(채팅방 참여자·최신 메시지) |

---

### 1.4 Race Condition (경쟁 조건)

| 항목 | 설명 |
|------|------|
| **정의** | 여러 트랜잭션이 동시에 같은 데이터를 읽고 수정할 때, 실행 순서에 따라 잘못된 결과가 나오는 현상 |
| **대표 사례** | Lost Update(읽기-수정-쓰기 사이 다른 트랜잭션 끼어듦), Check-Then-Act(조건 확인 후 행동 사이 끼어듦) |
| **발생 도메인** | Meetup(인원 초과), Payment(잔액·에스크로), Care(거래 확정), User(닉네임·경고), Chat(읽지 않은 수) |

---

### 1.5 비관적 락 (Pessimistic Lock)

| 항목 | 설명 |
|------|------|
| **정의** | 조회 시점에 DB 행에 락을 걸어, 다른 트랜잭션이 수정하지 못하게 하는 방식 |
| **JPA** | `@Lock(LockModeType.PESSIMISTIC_WRITE)` + `SELECT ... FOR UPDATE` |
| **특징** | 동시 접근 차단, 데이터 정합성 보장, 대기 시간 발생 가능 |
| **적용 도메인** | Payment(잔액 차감·에스크로), Care(거래 확정) |

---

### 1.6 원자적 UPDATE 쿼리 (Atomic Update Query)

| 항목 | 설명 |
|------|------|
| **정의** | `UPDATE ... SET col = col + 1 WHERE 조건`처럼 DB에서 읽기·수정·쓰기를 한 번에 수행하는 쿼리 |
| **특징** | Lock 없이 Race Condition 방지, 조건부 업데이트 가능 |
| **적용 예** | `incrementParticipantsIfAvailable`, `incrementUnreadCount`, `incrementWarningCount` |
| **적용 도메인** | Meetup, Chat, User, Board(댓글 수) |

---

### 1.7 DB 제약조건 (Unique, CHECK)

| 항목 | 설명 |
|------|------|
| **Unique** | 중복 데이터 방지 (닉네임, 에스크로-care_request 등) |
| **CHECK** | 도메인 규칙 강제 (잔액 ≥ 0, current_participants ≤ max_participants) |
| **역할** | 애플리케이션 로직을 우회하는 최종 방어선 |

## Part 2. 도메인별 상세 (원인 → 해결 → 결과)

---

### 2.1 Board 도메인

#### 원인

| 구분 | 내용 |
|------|------|
| **문제** | 게시글 목록 조회 시 작성자, 반응(좋아요/싫어요), 첨부파일을 각각 개별 쿼리로 조회 |
| **구조** | 1(게시글) + 100(작성자) + 100(좋아요) + 100(싫어요) + 100(첨부파일) = **301개 쿼리** (100개 게시글 기준) |
| **영향** | 745ms 지연, 22.5MB 메모리, DB 연결 풀 고갈 위험 |

#### 해결방법

| 단계 | 기법 | 구현 |
|------|------|------|
| 1 | **Fetch Join** | `JOIN FETCH b.user`로 작성자 정보 함께 조회 |
| 2 | **배치 조회** | `countByBoardsGroupByReactionType(boardIds)`로 반응 집계 |
| 3 | **배치 조회** | `getFilesByTargetIds(boardIds, BOARD)`로 첨부파일 일괄 조회 |
| 4 | **Map 매핑** | `Map<Long, ...>`로 메모리에서 DTO 매핑 |

#### 해결 후 결과

| 항목 | Before | After | 개선 |
|------|--------|-------|------|
| **쿼리 수** | 301개 | 3개 | **99% 감소** |
| **실행 시간** | 745ms | 30ms | **24.83배 개선** |
| **메모리** | 22.50 MB | 2 MB | **91% 감소** |

**Before 쿼리 예시**:
```
1. SELECT * FROM board ... (1)
2. SELECT * FROM users WHERE idx = ? (×100, 작성자)
3. SELECT COUNT(*) FROM board_reaction WHERE board_idx = ? AND reaction_type = 'LIKE' (×100)
4. SELECT COUNT(*) FROM board_reaction WHERE board_idx = ? AND reaction_type = 'DISLIKE' (×100)
5. SELECT * FROM attachment_file WHERE target_idx IN (...) AND target_type = 'BOARD' (×100 또는 N)
→ 총 301개
```

**After 쿼리 예시**:
```
1. SELECT b.*, u.* FROM board b JOIN users u ON ... (게시글+작성자)
2. SELECT board_idx, reaction_type, COUNT(*) FROM board_reaction WHERE board_idx IN (...) GROUP BY ...
3. SELECT * FROM attachment_file WHERE target_idx IN (...) AND target_type = 'BOARD'
→ 총 3개
```

### 2.2 User 도메인 (로그인 시 채팅방 목록)

#### 원인

| 구분 | 내용 |
|------|------|
| **문제** | 로그인 후 채팅방 목록 조회 시, 채팅방마다 참여자·최신 메시지를 개별 쿼리로 조회 |
| **구조** | 1(채팅방) + N×2(참여자) + N(메시지) = **3N+1개 쿼리** (채팅방 10개 → 31개, 실제 측정 21개) |
| **추가** | `conv.getMessages()`로 전체 메시지 로드 후 Java에서 최신 메시지 추출 → 메모리 낭비 |

#### 해결방법

| 단계 | 기법 | 구현 |
|------|------|------|
| 1 | **배치 조회** | `findParticipantsByConversationIdxsAndUserIdx(conversationIdxs, userId)` |
| 2 | **배치 조회** | `findParticipantsByConversationIdxsAndStatus(conversationIdxs, ACTIVE)` |
| 3 | **배치 조회** | `findLatestMessagesByConversationIdxs(conversationIdxs)` — 최신 메시지만 조회 |
| 4 | **Fetch Join** | 참여자 조회 시 `JOIN FETCH p.user` |
| 5 | **Map 매핑** | `Map<Long, ConversationParticipant>` 등으로 메모리 매핑 |

#### 해결 후 결과

| 항목 | Before | After | 개선 |
|------|--------|-------|------|
| **쿼리 수** | 21개 | 4개 | **80.95% 감소** |
| **실행 시간** | 305ms | 55ms | **81.97% 단축** |
| **메모리** | 0.58 MB | 0.13 MB | **77.24% 감소** |

**Before**:
```
1. SELECT * FROM conversation WHERE ...
2~3. SELECT * FROM conversationparticipant WHERE conversation_idx = 1 AND ...
4. SELECT * FROM chatmessage WHERE conversation_idx = 1 (전체 메시지)
5~6. conversation_idx = 2 ...
... (채팅방마다 반복)
```

**After**:
```
1. SELECT * FROM conversation WHERE ...
2. SELECT * FROM conversationparticipant WHERE conversation_idx IN (...) AND user_idx = ?
3. SELECT * FROM conversationparticipant WHERE conversation_idx IN (...) AND status = 'ACTIVE'
4. SELECT * FROM chatmessage WHERE conversation_idx IN (...) AND idx IN (SELECT MAX(...) ...)
```

### 2.3 Location 도메인

#### 원인

| 구분 | 내용 |
|------|------|
| **문제** | 초기 로드 시 전체 위치 서비스(약 22,000개)를 조회 후 프론트엔드에서 거리·필터링 |
| **영향** | DB 841ms, 네트워크 22MB, 프론트 78.9MB 메모리, 실제 표시는 10km 이내 약 100개 |

#### 해결방법

| 단계 | 기법 | 구현 |
|------|------|------|
| 1 | **위치 기반 검색** | 사용자 좌표 + 반경(10km) 전달 |
| 2 | **ST_Distance_Sphere** | MySQL 공간 함수로 DB 레벨 반경 필터링 |
| 3 | **백엔드 필터링** | 프론트엔드 거리 계산 제거, 백엔드에서 처리 |

```sql
SELECT * FROM locationservice
WHERE latitude IS NOT NULL AND longitude IS NOT NULL
  AND ST_Distance_Sphere(POINT(longitude, latitude), POINT(?, ?)) <= ?
ORDER BY rating DESC
```

#### 해결 후 결과

| 항목 | Before | After | 개선 |
|------|--------|-------|------|
| **조회 데이터 수** | 22,699개 | 1,026개 | **95.5% 감소** |
| **백엔드 처리 시간** | 885ms | 530ms | **40.1% 개선** |
| **프론트 전체 처리** | 1,484ms | 700ms | **52.8% 개선** |
| **네트워크 전송량** | 22 MB | 1 MB | **95.5% 감소** |
| **메모리(프론트)** | 78.90 MB | 28.6 MB | **63.8% 감소** |

### 2.4 Payment 도메인

#### 원인

| 구분 | 내용 |
|------|------|
| **문제 1** | 잔액 확인과 차감 사이에 다른 트랜잭션이 끼어들어 음수 잔액 발생 가능 |
| **문제 2** | 에스크로 상태 확인과 변경 사이에 중복 지급/환불 가능 |
| **시나리오** | 잔액 100, A가 80·B가 50 동시 차감 → 둘 다 확인 통과 → 음수 잔액 |

#### 해결방법

| 대상 | 기법 | 구현 |
|------|------|------|
| **잔액 차감** | 비관적 락 | `findByIdForUpdate(userIdx)` — `SELECT ... FOR UPDATE` |
| **에스크로 지급/환불** | 비관적 락 | `findByIdForUpdate(escrowIdx)`, `findByCareRequestForUpdate` |
| **최종 방어** | DB CHECK | `chk_pet_coin_balance: pet_coin_balance >= 0` |
| **중복 방지** | Unique | `PetCoinEscrow.careRequest` — `@JoinColumn(unique = true)` |

#### 해결 후 결과

| 항목 | Before | After |
|------|--------|-------|
| **잔액 차감** | Race Condition으로 음수 가능 | 비관적 락으로 순차 처리, 음수 방지 |
| **에스크로 지급/환불** | 중복 처리 가능 | 비관적 락으로 중복 지급/환불 방지 |
| **트랜잭션** | 에스크로 생성/지급 실패 시 부분 커밋 | 예외 재던지기로 전체 롤백 |

### 2.5 Meetup 도메인

#### 원인

| 구분 | 내용 |
|------|------|
| **문제** | `currentParticipants` 체크와 증가 사이에 다른 트랜잭션이 끼어들어 인원 초과 |
| **시나리오** | 최대 3명, 3명이 동시 참가 → 모두 `currentParticipants < maxParticipants` 통과 → 4명 참가 |

#### 해결방법

| 기법 | 구현 |
|------|------|
| **원자적 UPDATE** | `incrementParticipantsIfAvailable(meetupIdx)` — 조건부 증가를 DB에서 원자적으로 처리 |
| **DB 제약조건** | `CHECK (current_participants <= max_participants)` |

```java
@Modifying
@Query("UPDATE Meetup m SET m.currentParticipants = m.currentParticipants + 1 " +
       "WHERE m.idx = :meetupIdx AND m.currentParticipants < m.maxParticipants")
int incrementParticipantsIfAvailable(@Param("meetupIdx") Long meetupIdx);
```

#### 해결 후 결과

| 항목 | Before | After |
|------|--------|-------|
| **동시 참가** | 3명 시도 → 4명 참가 (초과) | 2명 성공, 1명 실패 → 정확히 3명 |
| **데이터 정합성** | Lost Update 발생 | 원자적 쿼리로 100% 보장 |

### 2.6 Care 도메인 (거래 확정)

#### 원인

| 구분 | 내용 |
|------|------|
| **문제** | 두 사용자가 동시에 '거래 확정' 시, 트랜잭션 격리로 상대방 변경을 못 보고 `allConfirmed`가 false → 후속 처리 미실행 |
| **결과** | DB상으로는 둘 다 확정인데, `CareRequest`는 OPEN 상태로 남는 Stuck State |

#### 해결방법

| 기법 | 구현 |
|------|------|
| **비관적 락** | `ConversationRepository.findByIdWithLock(conversationIdx)` |
| **순차 처리** | 한 명이 처리·커밋한 뒤 다른 한 명이 최신 데이터로 재조회 |

#### 해결 후 결과

| 항목 | Before | After |
|------|--------|-------|
| **동시 확정** | 후속 처리 스킵, OPEN 유지 | 순차 처리, IN_PROGRESS 정상 변경 |
| **TransientObjectException** | CareApplication 생성 시 발생 | `getReferenceById` + `saveAndFlush`로 해결 |

### 2.7 Chat 도메인 (읽지 않은 메시지 수)

#### 원인

| 구분 | 내용 |
|------|------|
| **문제** | 메시지 전송 시 `unreadCount`를 읽고 +1 후 저장하는 과정에서 Lost Update |

#### 해결방법

| 기법 | 구현 |
|------|------|
| **원자적 UPDATE** | `incrementUnreadCount(conversationIdx, senderUserId)` |

```java
@Modifying
@Query("UPDATE ConversationParticipant cp SET cp.unreadCount = cp.unreadCount + 1 " +
       "WHERE cp.conversation.idx = :conversationIdx AND cp.user.idx != :senderUserId")
void incrementUnreadCount(...);
```

#### 해결 후 결과

| 항목 | Before | After |
|------|--------|-------|
| **unreadCount** | Lost Update 가능 | 원자적 증가로 정확한 값 유지 |

### 2.8 User / Report / Board (중복 방지)

| 도메인 | 문제 | 해결 | 결과 |
|--------|------|------|------|
| **User** | 동시 가입 시 같은 닉네임 허용 | `@Column(unique = true)` + `DataIntegrityViolationException` 처리 | 중복 닉네임 방지 |
| **User** | 경고 횟수 증가 Lost Update | `incrementWarningCount` 원자적 쿼리 | 정확한 경고 수 |
| **Report** | 동시 신고 시 중복 신고 | `UNIQUE(target_type, target_idx, reporter_idx)` | 중복 신고 방지 |
| **Board** | 좋아요/싫어요 중복, 댓글 수 | Unique 제약 + 원자적 증가 | 중복·Lost Update 방지 |

## Part 3. 도메인별 요약표

| 도메인 | 원인 | 해결 | 전후 비교 |
|--------|------|------|-----------|
| **Board** | N+1 (작성자·반응·첨부파일) | Fetch Join + 배치 조회 | 301→3 쿼리, 745ms→30ms |
| **User(로그인)** | N+1 (채팅방 참여자·메시지) | 배치 조회 + Fetch Join | 21→4 쿼리, 305ms→55ms |
| **Location** | 전체 조회 후 클라이언트 필터링 | ST_Distance_Sphere 반경 검색 | 22,699→1,026건, 1.5초→0.7초 |
| **Payment** | 잔액·에스크로 Race Condition | 비관적 락 + DB 제약조건 | 음수·중복 지급/환불 방지 |
| **Meetup** | 인원 체크-증가 Race Condition | 원자적 UPDATE + CHECK | 인원 초과 방지 |
| **Care** | 거래 확정 Check-Then-Act | 비관적 락(Conversation) | Stuck State 해결 |
| **Chat** | unreadCount Lost Update | 원자적 UPDATE | 정확한 읽지 않은 수 |
