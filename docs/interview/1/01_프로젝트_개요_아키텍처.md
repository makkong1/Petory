# 1. 프로젝트 개요 및 아키텍처

## Q1-1. Petory 프로젝트를 간단히 소개해주세요.

### 답변 포인트
- 반려동물 통합 플랫폼으로 커뮤니티, 펫케어 매칭, 실종 동물 찾기, 위치 기반 서비스, 오프라인 모임 기능 제공
- 펫코인 결제(에스크로), 채팅, 알림, 신고/제재 시스템 포함
- 도메인 주도 설계(DDD)와 레이어드 아키텍처 적용
- 12개 도메인: User, Board, Care, Location, MissingPet, Meetup, Chat, Notification, Report, Statistics, File, Activity + Payment

### 상세 답변

#### 1. 핵심 기능
- **위치 기반 서비스**: ST_Distance_Sphere 반경 검색, Haversine 거리 계산, 네이버맵 API 연동
- **펫케어 & 매칭**: 요청→지원→매칭→케어→리뷰, 채팅 연동, 펫코인 에스크로 결제
- **산책 & 모임**: 반경 검색, 인원 관리, 채팅방 자동 생성
- **커뮤니티 & 실종 제보**: 블라인드 처리, 인기글 스냅샷, Magazine 레이아웃
- **관리자 대시보드**: DailyStatistics 일별 집계, Recharts 시각화
- **알림**: Redis+MySQL 이중 저장소, SSE 실시간 푸시
- **신고 & 제재**: 경고 누적 자동 이용제한, 스케줄러 만료 해제

#### 2. 기술 스택
- **Backend**: Spring Boot 3.5.7, Java 17, MySQL 8.0, Redis, JWT, OAuth2
- **Frontend**: React 19, Styled-components, Axios, Recharts
- **실시간**: WebSocket(STOMP), SSE

## Q1-2. 프로젝트에서 도메인 주도 설계(DDD)를 어떻게 적용했나요?

### 답변 포인트
- 도메인별 패키지 구조로 응집도 향상
- Repository + Jpa*Adapter 패턴으로 데이터 액세스 추상화
- 도메인별 구체 예외로 명확한 에러 처리
- 도메인 간 느슨한 결합

### 상세 답변

#### 1. 도메인 패키지 구조
**위치**: `domain/*/`

```
domain/board/
  ├── controller/     # BoardController, MissingPetBoardController
  ├── service/        # BoardService, CommentService, BoardPopularityScheduler
  ├── entity/         # Board, Comment, BoardReaction
  ├── repository/     # BoardRepository, JpaBoardAdapter, SpringDataJpaBoardRepository
  ├── exception/      # BoardNotFoundException, BoardConflictException
  ├── dto/
  └── converter/
```

#### 2. Repository 패턴
- **도메인 인터페이스**: `BoardRepository` - 도메인에 필요한 메서드만 노출
- **JpaBoardAdapter**: Spring Data JPA 구현체 래핑
- **SpringDataJpaBoardRepository**: JpaRepository 상속, 쿼리 메서드, @Query

#### 3. 도메인별 예외
- `BoardNotFoundException`, `BoardConflictException`
- `UserNotFoundException`, `PetNotFoundException`
- `FileValidationException`, `FileUploadValidationException`, `FileNotFoundException`
- `MeetupNotFoundException`, `MeetupConflictException`
- `PaymentConflictException`, `PaymentValidationException`

## Q1-3. 레이어드 아키텍처의 각 레이어 역할을 설명해주세요.

### 답변 포인트
- **Controller**: HTTP 요청/응답, REST API, WebSocket, SSE 엔드포인트
- **Service**: 비즈니스 로직, 트랜잭션 관리
- **Repository**: 데이터 액세스 추상화
- **Entity**: 도메인 모델

### 상세 답변

#### 1. Controller Layer
**역할**: HTTP 요청/응답 처리

**구성**:
- REST API: `@RestController`, `@GetMapping`, `@PostMapping` 등
- WebSocket: `ChatWebSocketController` (STOMP)
- SSE: `NotificationController.subscribe()`
- Admin: `AdminStatisticsController`, `AdminFileController` 등

#### 2. Service Layer
**역할**: 비즈니스 로직, 트랜잭션 경계

**특징**:
- `@Transactional(readOnly = true)` 클래스 레벨 기본값
- 쓰기 작업에 `@Transactional` 명시
- Repository를 통한 데이터 액세스

#### 3. Repository Layer
**역할**: 데이터 액세스 추상화

**구성**:
- `BoardRepository` (인터페이스)
- `JpaBoardAdapter` (구현체)
- `SpringDataJpaBoardRepository` (JPA)

#### 4. Entity Layer
**역할**: 도메인 모델 표현

**예시**: Board, Users, CareRequest, Meetup, PetCoinEscrow 등

## Q1-4. 프로젝트의 핵심 성과를 설명해주세요.

> **상세 문서**: [01_Q1-4_핵심성과_상세.md](./01_Q1-4_핵심성과_상세.md) — 원인·해결·결과(전후 비교), 공통 개념·정의 정리

### 답변 포인트
- **Board**: 301개 쿼리 → 3개 (99% 감소), 745ms → 30ms (24.83배)
- **Location**: 22,699개 → 1,026개 (95.5% 감소)
- **User**: 로그인 쿼리 21개 → 4개 (80.95% 감소)
- **Payment**: 비관적 락으로 에스크로 Race Condition 방지

### 상세 답변

#### 1. Board 도메인
- **원인**: 게시글 목록 조회 시 작성자·반응·첨부파일을 각각 개별 쿼리로 조회 (N+1)
- **해결**: Fetch Join(작성자) + 배치 조회(반응, 첨부파일) + Map 매핑
- **결과**: 301개 → 3개 쿼리 (99% 감소), 745ms → 30ms (24.83배), 메모리 22.5MB → 2MB (91% 감소)

#### 2. Location 도메인
- **원인**: 초기 로드 시 전체 22,000개 조회 후 프론트엔드에서 거리·필터링
- **해결**: ST_Distance_Sphere로 사용자 위치 기준 10km 반경 검색 (백엔드 필터링)
- **결과**: 22,699개 → 1,026개 (95.5% 감소), 1.5초 → 0.7초, 네트워크 22MB → 1MB

#### 3. User 도메인 (로그인)
- **원인**: 채팅방마다 참여자·최신 메시지를 개별 쿼리로 조회, 전체 메시지 로드
- **해결**: 채팅방 ID 목록 추출 → 참여자·최신 메시지 배치 조회 + Fetch Join
- **결과**: 21개 → 4개 쿼리 (80.95% 감소), 305ms → 55ms (81.97% 단축), 메모리 0.58MB → 0.13MB

#### 4. Payment 도메인
- **원인**: 잔액 확인-차감, 에스크로 상태 확인-변경 사이 Race Condition → 음수 잔액·중복 지급/환불
- **해결**: `findByIdForUpdate`, `findByCareRequestForUpdate` 비관적 락 + DB CHECK 제약조건
- **결과**: 잔액 차감·에스크로 상태 변경·중복 지급/환불 방지, 트랜잭션 롤백 처리

#### 5. 기타 도메인 (동시성·중복 방지)
- **Meetup**: 원자적 UPDATE(`incrementParticipantsIfAvailable`) + CHECK 제약 → 인원 초과 방지
- **Care**: 비관적 락(Conversation) → 거래 확정 Stuck State 해결
- **Chat**: 원자적 UPDATE(`incrementUnreadCount`) → 읽지 않은 수 Lost Update 방지
- **User/Report/Board**: DB Unique 제약 + 원자적 증가 쿼리 → 중복·Lost Update 방지
