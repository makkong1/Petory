# 9. 트러블슈팅 & 문제 해결

## Q9-1. 로그인 시 N+1 문제를 어떻게 해결했나요?

### 답변 포인트
- **문제**: 로그인 시 채팅방 목록 조회 과정에서 N+1 문제 발생
- **해결**: 배치 조회 패턴, Fetch Join, 최신 메시지만 조회
- **결과**: 쿼리 수 21개 → 4개 (80.95% 감소)

### 상세 답변

#### 1. 문제 상황
**위치**: `domain/chat/service/ChatMessageService.java` (최적화 전)

**Before 최적화**:
```
로그인 성공
  ↓
사용자의 채팅방 목록 조회: 1개 쿼리
  ↓
각 채팅방의 참여자 조회: 10개 쿼리 (N+1)
  ↓
각 채팅방의 최신 메시지 조회: 10개 쿼리 (N+1)
  ↓
총 21개 쿼리
```

#### 2. 해결 방법

##### 2-1. Fetch Join으로 참여자 정보 조회
**위치**: `domain/chat/repository/ConversationRepository.java`

**코드 예시**:
```java
// domain/chat/repository/ConversationRepository.java
@Query("SELECT DISTINCT c FROM Conversation c " +
       "JOIN FETCH c.participants p " +
       "WHERE p.user.idx = :userId AND p.status = 'ACTIVE'")
List<Conversation> findByUserId(@Param("userId") Long userId);
```

**효과**: 10개 쿼리 → 1개 쿼리

##### 2-2. 배치 조회로 최신 메시지 조회
**위치**: `domain/chat/service/ChatMessageService.java`

**전체 흐름**:
```
채팅방 목록 조회 완료
  ↓
채팅방 ID 리스트 추출
  ↓
IN 절로 최신 메시지 배치 조회
  ↓
1개 쿼리로 모든 최신 메시지 조회
```

**코드 예시**:
```java
// domain/chat/service/ChatMessageService.java
List<Long> conversationIds = conversations.stream()
    .map(Conversation::getIdx)
    .collect(Collectors.toList());

// 배치 조회: 최신 메시지
Map<Long, ChatMessage> latestMessagesMap = chatMessageRepository
    .findLatestMessagesByConversationIds(conversationIds)
    .stream()
    .collect(Collectors.toMap(
        msg -> msg.getConversation().getIdx(),
        msg -> msg
    ));
```

**효과**: 10개 쿼리 → 1개 쿼리

##### 2-3. 읽지 않은 메시지 수 배치 조회
**위치**: `domain/chat/repository/ConversationParticipantRepository.java`

**코드 예시**:
```java
// 배치 조회: 읽지 않은 메시지 수
Map<Long, Integer> unreadCountMap = participantRepository
    .findUnreadCountsByConversationIds(conversationIds, userId)
    .stream()
    .collect(Collectors.toMap(
        ConversationParticipant::getConversationIdx,
        ConversationParticipant::getUnreadCount
    ));
```

**효과**: 10개 쿼리 → 1개 쿼리

#### 3. 최종 결과
**After 최적화**:
```
1. 채팅방 + 참여자 조회 (Fetch Join): 1개 쿼리
2. 최신 메시지 배치 조회: 1개 쿼리
3. 읽지 않은 메시지 수 배치 조회: 1개 쿼리
4. 사용자 정보 조회: 1개 쿼리

총 4개 쿼리
```

**성능 개선**:
- 쿼리 수: 21개 → 4개 (**80.95% 감소**)

## Q9-2. 제재 시스템 동시성 문제를 어떻게 해결했나요?

### 답변 포인트
- **문제**: 여러 관리자가 동시에 경고 부여 시 Lost Update 발생
- **해결**: DB 레벨에서 원자적 증가 쿼리 사용
- **효과**: 경고 횟수가 정확하게 증가하며, 자동 이용제한이 정확히 한 번만 적용

### 상세 답변

#### 1. 문제 상황
**시나리오**: 관리자 A와 B가 동시에 같은 사용자에게 경고 부여

**Before 해결**:
```
시간 T1: 관리자 A가 warningCount 읽기 (현재: 2)
시간 T2: 관리자 B가 warningCount 읽기 (현재: 2)
시간 T3: 관리자 A가 warningCount = 3으로 저장
시간 T4: 관리자 B가 warningCount = 3으로 저장

결과: warningCount = 3 (실제로는 4가 되어야 함)
```

**문제점**:
- Lost Update 발생
- 경고 횟수 누락
- 자동 이용제한 미적용 또는 중복 적용

#### 2. 해결 방법
**위치**: `domain/user/repository/SpringDataJpaUsersRepository.java`
**메서드**: `incrementWarningCount()`

**코드 예시**:
```java
// domain/user/repository/SpringDataJpaUsersRepository.java
@Modifying
@Query("UPDATE Users u SET u.warningCount = u.warningCount + 1 WHERE u.idx = :userId")
int incrementWarningCount(@Param("userId") Long userId);
```

**사용 예시**:
```java
// domain/user/service/UserSanctionService.java
@Transactional
public UserSanction addWarning(Long userId, String reason, Long adminId, Long reportId) {
    // 경고 추가
    UserSanction warning = UserSanction.builder()
        .user(user)
        .sanctionType(UserSanction.SanctionType.WARNING)
        .reason(reason)
        .build();
    sanctionRepository.save(warning);
    
    // 경고 횟수 원자적 증가 (동시성 문제 해결)
    usersRepository.incrementWarningCount(userId);
    
    // 업데이트된 사용자 정보 다시 조회
    user = usersRepository.findById(userId).orElseThrow();
    
    // 경고 3회 이상이면 자동 이용제한
    if (user.getWarningCount() >= WARNING_THRESHOLD) {
        addSuspension(userId, "경고 누적으로 인한 자동 이용제한", adminId, reportId, 3);
    }
    
    return warning;
}
```

#### 3. 동작 원리
**DB 레벨 원자성**:
```
UPDATE users SET warning_count = warning_count + 1 WHERE idx = ?
```

- DB 엔진이 내부적으로 락 관리
- 여러 요청이 동시에 들어와도 순차 처리
- Lost Update 문제 완전 해결

#### 4. 효과
- 경고 횟수가 정확하게 증가
- 자동 이용제한이 정확히 한 번만 적용
- 추가 락 메커니즘 불필요

## Q9-3. 펫케어 요청 확정 시 Race Condition을 어떻게 해결했나요?

### 답변 포인트
- 트랜잭션 내에서 상태 확인 및 변경
- 원자적 연산으로 중복 확정 방지
- DB 제약조건 활용

### 상세 답변

#### 1. 문제 상황
**시나리오**: 펫케어 요청에 동시에 여러 명이 확정 요청

**Before 해결**:
```
사용자 A: 요청 상태 확인 (PENDING)
사용자 B: 요청 상태 확인 (PENDING)
사용자 A: 상태 변경 (CONFIRMED)
사용자 B: 상태 변경 (CONFIRMED) - 중복 확정
```

#### 2. 해결 방법
**위치**: `domain/care/service/CareRequestService.java`

**전체 흐름**:
```
펫케어 확정 요청
  ↓
@Transactional 트랜잭션 시작
  ↓
요청 상태 확인 (PENDING만 허용)
  ↓
상태 변경 (CONFIRMED)
  ↓
서비스 제공자 연결
  ↓
트랜잭션 커밋
```

**코드 예시**:
```java
// domain/care/service/CareRequestService.java
@Transactional
public CareRequestDTO confirmRequest(Long requestId, Long providerId) {
    CareRequest request = careRequestRepository.findById(requestId)
        .orElseThrow(() -> new IllegalArgumentException("요청을 찾을 수 없습니다."));
    
    // 트랜잭션 내에서 상태 확인
    if (request.getStatus() != CareRequestStatus.PENDING) {
        throw new IllegalStateException("이미 처리된 요청입니다.");
    }
    
    Users provider = usersRepository.findById(providerId)
        .orElseThrow(() -> new IllegalArgumentException("서비스 제공자를 찾을 수 없습니다."));
    
    // 상태 변경
    request.setStatus(CareRequestStatus.CONFIRMED);
    request.setProvider(provider);
    request.setConfirmedAt(LocalDateTime.now());
    
    careRequestRepository.save(request);
    
    return careRequestConverter.toDTO(request);
}
```

#### 3. 트랜잭션 격리 수준
**기본값**: READ_COMMITTED

**동작 방식**:
- 트랜잭션 A가 상태 변경 중일 때
- 트랜잭션 B는 커밋된 데이터만 읽음
- 트랜잭션 A 커밋 후 트랜잭션 B가 읽으면 CONFIRMED 상태
- 중복 확정 방지

#### 4. 효과
- Race Condition 방지
- 중복 확정 방지
- 데이터 일관성 유지

## Q9-4. 모임 참여 시 인원 초과 문제를 어떻게 해결했나요?

### 답변 포인트
- 트랜잭션 내에서 인원 수 확인 및 증가
- DB 제약조건으로 최대 인원 초과 방지
- Race Condition 방지

### 상세 답변

#### 1. 문제 상황
**시나리오**: 모임 최대 인원 10명, 현재 9명일 때 동시에 2명이 참여

**Before 해결**:
```
사용자 A: 현재 인원 확인 (9명)
사용자 B: 현재 인원 확인 (9명)
사용자 A: 참여 (10명)
사용자 B: 참여 (11명) - 최대 인원 초과
```

#### 2. 해결 방법
**위치**: `domain/meetup/service/MeetupService.java`

**전체 흐름**:
```
모임 참여 요청
  ↓
@Transactional 트랜잭션 시작
  ↓
모임 정보 조회
  ↓
인원 수 확인 (최대 인원 체크)
  ↓
인원 수 증가
  ↓
참여자 추가
  ↓
트랜잭션 커밋
```

**코드 예시**:
```java
// domain/meetup/service/MeetupService.java
@Transactional
public void joinMeetup(Long meetupId, Long userId) {
    Meetup meetup = meetupRepository.findById(meetupId)
        .orElseThrow(() -> new IllegalArgumentException("모임을 찾을 수 없습니다."));
    
    Users user = usersRepository.findById(userId)
        .orElseThrow(() -> new IllegalArgumentException("사용자를 찾을 수 없습니다."));
    
    // 트랜잭션 내에서 인원 수 확인
    if (meetup.getCurrentParticipants() >= meetup.getMaxParticipants()) {
        throw new IllegalStateException("모임 인원이 가득 찼습니다.");
    }
    
    // 이미 참여 중인지 확인
    boolean alreadyJoined = meetupParticipantRepository
        .existsByMeetupIdxAndUserIdx(meetupId, userId);
    if (alreadyJoined) {
        throw new IllegalStateException("이미 참여 중인 모임입니다.");
    }
    
    // 인원 수 증가
    meetup.setCurrentParticipants(meetup.getCurrentParticipants() + 1);
    meetupRepository.save(meetup);
    
    // 참여자 추가
    MeetupParticipant participant = MeetupParticipant.builder()
        .meetup(meetup)
        .user(user)
        .build();
    meetupParticipantRepository.save(participant);
}
```

#### 3. 추가 보호 장치
**DB 제약조건**:
```sql
-- 모임 엔티티에 CHECK 제약조건 (선택적)
ALTER TABLE meetup 
ADD CONSTRAINT chk_participants 
CHECK (current_participants <= max_participants);
```

#### 4. 효과
- 최대 인원 초과 방지
- 동시성 문제 해결
- Race Condition 방지

## Q9-5. 펫코인 에스크로 Race Condition을 어떻게 해결했나요?

### 답변 포인트
- **문제**: 거래 확정·완료·취소 시 동시 요청으로 중복 지급/환불 가능
- **해결**: 비관적 락 (`@Lock(PESSIMISTIC_WRITE)`) - `findByIdForUpdate`, `findByCareRequestForUpdate`
- **효과**: 잔액 차감·에스크로 상태 변경·지급/환불 시 Race Condition 방지

### 상세 답변

**위치**: `domain/payment/repository/`, `PetCoinEscrowService`, `PetCoinService`

**전체 흐름**:
```
거래 확정/완료/취소 요청
  ↓
findByIdForUpdate() 또는 findByCareRequestForUpdate() - 비관적 락 획득
  ↓
상태 확인 후 잔액 차감/지급/환불
  ↓
트랜잭션 커밋 시 락 해제
```

## 📝 핵심 정리

### 로그인 N+1 문제
- **Fetch Join**: 채팅방 + 참여자 함께 조회
- **배치 조회**: 최신 메시지, 읽지 않은 수 IN 절 조회
- **결과**: 21개 → 4개 쿼리 (80.95% 감소)

### 제재 시스템 동시성
- **원자적 증가 쿼리**: DB 레벨에서 처리
- **위치**: `UsersRepository.incrementWarningCount()`
- **효과**: Lost Update 완전 해결

### 펫케어 확정 Race Condition
- **트랜잭션 내 확인**: 상태 확인 및 변경
- **격리 수준**: READ_COMMITTED
- **효과**: 중복 확정 방지

### 모임 참여 인원 초과
- **원자적 UPDATE**: `incrementParticipantsIfAvailable()` - WHERE 조건으로 인원 초과 시 0행
- **효과**: 최대 인원 초과 방지

### 펫코인 에스크로 Race Condition
- **비관적 락**: `findByIdForUpdate`, `findByCareRequestForUpdate` - 잔액 차감·에스크로 상태 변경 시
- **효과**: 중복 지급/환불 방지
