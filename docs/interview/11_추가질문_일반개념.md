## 1. 추가 질문 대비 (후속 질문)

### N+1 / Fetch Join 관련

| 후속 질문 | 답변 포인트 |
|----------|-------------|
| **왜 Fetch Join이 아니라 배치 조회로 했나요?** | Fetch Join은 부모-직접연관(1:1, N:1)에 적합. 반응·첨부파일은 1:N이고 여러 부모 ID에 대해 조회하므로 IN 절 배치가 더 적합. Fetch Join으로 1:N을 하면 카테시안 곱으로 행 수 폭증. |
| **배치 크기 500으로 한 이유는?** | IN 절이 너무 크면 DB 성능 저하·쿼리 제한 초과 가능. 500~1000개가 일반적 권장. 우리는 500으로 설정. |
| **Fetch Join과 @EntityGraph 차이는?** | @EntityGraph도 연관 로딩 지정 가능. Fetch Join은 JPQL에서 명시적으로 제어 가능하고, WHERE 조건에 연관 엔티티 조건 추가 가능. 우리는 JPQL @Query로 Fetch Join 사용. |

### 동시성 관련

| 후속 질문 | 답변 포인트 |
|----------|-------------|
| **왜 비관적 락을 썼나요? 낙관적 락은?** | 펫코인 잔액·에스크로는 충돌 시 재시도보다 즉시 일관성 보장이 중요. 비관적 락이 적합. 낙관적 락은 읽기 많고 충돌 적을 때 유리. |
| **모임 참여는 원자적 UPDATE만 쓰는데, 비관적 락은 안 쓰나요?** | `incrementParticipantsIfAvailable`는 UPDATE ... WHERE currentParticipants < maxParticipants로 DB가 원자적으로 처리. 단일 쿼리로 충돌 방지 가능해 별도 락 불필요. |
| **경고 횟수는 왜 원자적 증가 쿼리인가요?** | 여러 관리자가 동시 경고 시 Lost Update 방지. `UPDATE users SET warning_count = warning_count + 1`로 DB 레벨 원자성 확보. |

### 캐시 관련

| 후속 질문 | 답변 포인트 |
|----------|-------------|
| **boardDetail 캐시를 제거한 이유는?** | 조회수를 실시간 반영해야 해서. @Cacheable이면 캐시 만료 전까지 조회수 갱신이 안 됨. 실시간성 우선으로 제거. |
| **Redis TTL 24시간 기준은?** | 알림은 최신 50개만 자주 조회되고, 오래된 건 DB에서 조회. 24시간이면 실시간 조회 부담을 줄이면서 메모리도 관리 가능. |

### 아키텍처 관련

| 후속 질문 | 답변 포인트 |
|----------|-------------|
| **왜 Repository + Adapter 패턴인가요?** | 도메인이 JPA에 직접 의존하지 않게 하려고. 도메인 인터페이스만 두고, Adapter가 Spring Data JPA 구현을 감쌈. 테스트·교체 용이. |
| **도메인별 예외를 쓰는 이유는?** | GlobalExceptionHandler에서 예외별 HTTP 상태·메시지 분기 가능. IllegalArgumentException보다 명확하고, 클라이언트가 errorCode로 처리하기 좋음. |

## 2. 일반 개념 (프로젝트에 없지만 면접 대비)

### 낙관적 락 vs 비관적 락

| 구분 | 낙관적 락 (Optimistic Lock) | 비관적 락 (Pessimistic Lock) |
|------|----------------------------|------------------------------|
| **방식** | @Version 필드로 버전 관리, 수정 시 버전 비교 | SELECT ... FOR UPDATE로 락 획득 |
| **충돌 시** | OptimisticLockException → 재시도 | 락 해제될 때까지 대기 |
| **적합한 경우** | 읽기 많고, 충돌 적을 때 | 쓰기 비중 높고, 충돌 시 즉시 일관성 필요할 때 |
| **우리 프로젝트** | 미사용 | 펫코인 에스크로 (findByIdForUpdate) |

### 분산 트랜잭션

- **문제**: 여러 DB/서비스에 걸친 작업의 원자성 보장
- **해결**: 2PC, Saga, Outbox 패턴 등
- **우리 프로젝트**: 모놀리식 + 단일 DB. 분산 트랜잭션 미사용. 확장 시 Saga(이벤트 기반 보상) 고려 가능.

### CAP 정리

- **Consistency**: 모든 노드가 같은 데이터
- **Availability**: 요청에 항상 응답
- **Partition tolerance**: 네트워크 분할 시에도 동작
- **정리**: P가 있으면 C와 A 중 하나만 만족 가능
- **우리 프로젝트**: 단일 DB로 CP 지향. Redis 캐시는 최종 일관성 수준.

### 인덱스 전략

| 인덱스 종류 | 용도 | 우리 프로젝트 예시 |
|------------|------|-------------------|
| **B-Tree** | 일반 조회, 범위 검색 | (user_idx, is_deleted, created_at) |
| **UNIQUE** | 중복 방지 | email, provider+providerId |
| **SPATIAL** | 위치 검색 | LocationService (POINT) |
| **FULLTEXT** | 텍스트 검색 | (선택) 게시글 제목·내용 |

### 트랜잭션 격리 수준

| 수준 | Dirty Read | Non-repeatable Read | Phantom Read |
|------|------------|---------------------|--------------|
| READ_UNCOMMITTED | O | O | O |
| READ_COMMITTED | X | O | O |
| REPEATABLE_READ | X | X | O (Gap Lock로 일부 방지) |
| SERIALIZABLE | X | X | X |

- **MySQL 기본값**: REPEATABLE_READ
- **우리 프로젝트**: 기본값 사용. 펫코인 등 민감 구간은 비관적 락으로 보완.

### 메시지 큐 (RabbitMQ, Kafka)

- **용도**: 비동기 처리, 서비스 간 느슨한 결합, 이벤트 드리븐
- **우리 프로젝트**: 미사용. 동기 처리. 확장 시 알림 발송·통계 집계 등을 큐로 분리 가능.

### CQRS (Command Query Responsibility Segregation)

- **개념**: 쓰기(Command)와 읽기(Query) 모델 분리
- **우리 프로젝트**: 미적용. 단일 모델. 통계는 DailyStatistics로 사전 집계해 읽기 부하 분산.
