# 7. 실시간 통신

## Q7-1. WebSocket과 SSE의 차이를 설명해주세요.

### 답변 포인트
- **WebSocket**: 양방향 통신, STOMP 프로토콜 사용, 채팅에 활용
- **SSE**: 서버 → 클라이언트 단방향, EventSource API, 알림에 활용
- 프로젝트에서 WebSocket은 채팅, SSE는 알림에 사용

### 상세 답변

#### 1. WebSocket 특징
**용도**: 실시간 양방향 통신

**특징**:
- 양방향 통신 (클라이언트 ↔ 서버)
- 지속적인 연결 유지
- 낮은 지연 시간
- STOMP 프로토콜 사용

**프로젝트 사용**: 채팅 시스템

#### 2. SSE 특징
**용도**: 서버 → 클라이언트 단방향 통신

**특징**:
- 단방향 통신 (서버 → 클라이언트)
- HTTP 기반 (WebSocket보다 간단)
- 자동 재연결 지원
- EventSource API 사용

**프로젝트 사용**: 알림 시스템

#### 3. 비교표

| 항목 | WebSocket | SSE |
|------|-----------|-----|
| 통신 방향 | 양방향 | 단방향 (서버→클라이언트) |
| 프로토콜 | STOMP | HTTP |
| 연결 유지 | 지속적 | 지속적 |
| 사용 용도 | 채팅 | 알림 |
| 구현 복잡도 | 높음 | 낮음 |

## Q7-2. WebSocket(STOMP) 기반 채팅 시스템을 설명해주세요.

### 답변 포인트
- STOMP 프로토콜로 메시지 브로커 활용
- 1:1 채팅, 그룹 채팅 지원
- 펫케어/모임 생성 시 채팅방 자동 연동
- 읽음 상태 관리

### 상세 답변

#### 1. WebSocket 설정
**위치**: `global/websocket/config/WebSocketConfig.java`

**전체 흐름**:
```
WebSocket 연결 요청
  ↓
WebSocketConfig.registerStompEndpoints()
  ↓
엔드포인트 등록 (/ws, /chat)
  ↓
인증 인터셉터 추가
  ↓
SockJS 지원
```

**코드 예시**:
```java
// global/websocket/config/WebSocketConfig.java
@Override
public void registerStompEndpoints(StompEndpointRegistry registry) {
    registry.addEndpoint("/ws", "/chat")
        .setAllowedOriginPatterns("*")
        .addInterceptors(authenticationInterceptor)
        .setHandshakeHandler(new WebSocketHandshakeHandler())
        .withSockJS();
}

@Override
public void configureMessageBroker(MessageBrokerRegistry registry) {
    // 서버 → 클라이언트 (구독 경로)
    registry.enableSimpleBroker("/topic", "/queue", "/user");
    
    // 클라이언트 → 서버 (전송 경로)
    registry.setApplicationDestinationPrefixes("/app");
    
    // 사용자별 개인 메시지 프리픽스
    registry.setUserDestinationPrefix("/user");
}
```

#### 2. 채팅 메시지 전송
**위치**: `domain/chat/controller/ChatWebSocketController` 또는 REST `ChatController`

**전체 흐름**:
```
클라이언트 메시지 전송
  ↓
ChatController.sendMessage()
  ↓
ChatMessageService.saveMessage()
  ↓
메시지 DB 저장
  ↓
SimpMessagingTemplate으로 구독자에게 전송
```

**코드 예시**:
```java
// domain/chat/controller/ChatWebSocketController.java
@MessageMapping("/chat.send")
public void sendMessage(@Payload ChatMessageDTO messageDTO, Principal principal) {
    ChatMessage savedMessage = chatMessageService.saveMessage(
        messageDTO.getConversationId(),
        Long.parseLong(principal.getName()),
        messageDTO.getContent()
    );
    
    // 구독자에게 메시지 전송
    messagingTemplate.convertAndSend(
        "/topic/conversation/" + messageDTO.getConversationId(),
        chatMessageConverter.toDTO(savedMessage)
    );
}
```

#### 3. 채팅방 자동 연동
**위치**: `domain/care/service/CareRequestService.java`, `domain/meetup/service/MeetupService.java`

**전체 흐름**:
```
펫케어/모임 생성
  ↓
CareRequestService.createCareRequest() / MeetupService.createMeetup()
  ↓
ConversationService.createConversation() / createGroupConversation()
  ↓
채팅방(Conversation) 자동 생성 및 참여자 추가
```

**코드 예시**:
```java
// domain/care/service/CareRequestService.java
@Transactional
public CareRequestDTO createCareRequest(CareRequestDTO dto) {
    CareRequest saved = careRequestRepository.save(careRequest);
    // ConversationService.createConversation() 또는 이벤트로 채팅방 자동 생성
    return careRequestConverter.toDTO(saved);
}
```

#### 4. 읽음 상태 관리
**위치**: `domain/chat/service/ChatMessageService.java`, `ConversationParticipant`

**전체 흐름**:
```
메시지 조회
  ↓
읽음 상태 업데이트 (ConversationParticipant 등)
  ↓
읽음 상태 변경 알림 전송
```

## Q7-3. SSE 기반 알림 시스템을 설명해주세요.

### 답변 포인트
- 이중 저장소 전략: Redis + MySQL
- Redis에 최신 50개 알림 저장 (TTL 24시간)
- SSE로 실시간 알림 푸시
- 읽음 처리 및 알림 이력 관리

### 상세 답변

#### 1. SSE 설정
**위치**: `domain/notification/controller/NotificationController.java`

**전체 흐름**:
```
SSE 연결 요청
  ↓
NotificationController.subscribe()
  ↓
SseEmitter 생성
  ↓
NotificationSseService에 등록
  ↓
연결 유지 및 알림 전송
```

**코드 예시**:
```java
// domain/notification/controller/NotificationController.java
@GetMapping(value = "/subscribe", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
public SseEmitter subscribe(@RequestParam String token) {
    // 토큰에서 사용자 ID 추출
    Long userId = jwtUtil.getIdFromToken(token);
    
    // SSE 연결 생성
    SseEmitter emitter = new SseEmitter(Long.MAX_VALUE);
    
    // 알림 서비스에 등록
    notificationSseService.addEmitter(userId, emitter);
    
    // 연결 종료 시 정리
    emitter.onCompletion(() -> notificationSseService.removeEmitter(userId, emitter));
    emitter.onTimeout(() -> notificationSseService.removeEmitter(userId, emitter));
    
    return emitter;
}
```

#### 2. 이중 저장소 전략
**위치**: `domain/notification/service/NotificationService.java`

**전체 흐름**:
```
알림 생성
  ↓
MySQL에 저장 (영구 저장)
  ↓
Redis에 최신 50개 저장 (TTL 24시간)
  ↓
SSE로 실시간 전송
```

**코드 예시**:
```java
// domain/notification/service/NotificationService.java
@Transactional
public NotificationDTO createNotification(Long userId, NotificationType type, 
        String title, String content, Long relatedId, String relatedType) {
    
    // MySQL에 저장
    Notification notification = Notification.builder()
        .user(user)
        .type(type)
        .title(title)
        .content(content)
        .build();
    Notification saved = notificationRepository.save(notification);
    
    NotificationDTO dto = notificationConverter.toDTO(saved);
    
    // Redis에 실시간 알림 저장 (최신 알림 목록 관리)
    saveToRedis(userId, dto);
    
    // SSE를 통해 실시간 알림 전송
    sseService.sendNotification(userId, dto);
    
    return dto;
}
```

#### 3. Redis 저장 로직
**위치**: `domain/notification/service/NotificationService.java`
**메서드**: `saveToRedis()`

**전체 흐름**:
```
알림 생성
  ↓
Redis에서 기존 알림 목록 조회
  ↓
새 알림 추가
  ↓
최신 50개만 유지
  ↓
TTL 24시간 설정
```

**코드 예시**:
```java
// domain/notification/service/NotificationService.java
private void saveToRedis(Long userId, NotificationDTO notification) {
    String key = REDIS_KEY_PREFIX + userId;
    
    // 기존 알림 목록 조회
    List<NotificationDTO> notifications = getFromRedis(userId);
    if (notifications == null) {
        notifications = new ArrayList<>();
    }
    
    // 새 알림 추가
    notifications.add(0, notification);
    
    // 최신 50개만 유지
    if (notifications.size() > 50) {
        notifications = notifications.subList(0, 50);
    }
    
    // Redis에 저장 (TTL 24시간)
    notificationRedisTemplate.opsForValue().set(
        key, 
        notifications, 
        REDIS_TTL_HOURS, 
        TimeUnit.HOURS
    );
}
```

#### 4. SSE 알림 전송
**위치**: `domain/notification/service/NotificationSseService.java`

**전체 흐름**:
```
알림 생성
  ↓
NotificationSseService.sendNotification()
  ↓
해당 사용자의 SseEmitter 조회
  ↓
메시지 전송
  ↓
연결 끊어졌으면 제거
```

**코드 예시**:
```java
// domain/notification/service/NotificationSseService.java
public void sendNotification(Long userId, NotificationDTO notification) {
    SseEmitter emitter = emitters.get(userId);
    
    if (emitter != null) {
        try {
            emitter.send(SseEmitter.event()
                .name("notification")
                .data(notification));
        } catch (IOException e) {
            // 연결 끊어졌으면 제거
            removeEmitter(userId, emitter);
        }
    }
}
```

## Q7-4. 채팅 읽음 상태를 어떻게 최적화했나요?

### 답변 포인트
- 배치 조회로 읽음 상태 정보 한 번에 조회
- Redis 캐싱 활용
- N+1 문제 해결

### 상세 답변

#### 1. 읽음 상태 조회 최적화
**위치**: `domain/chat/service/ChatMessageService.java`

**Before 최적화**:
```
각 메시지마다 읽음 상태 조회
  ↓
N개 쿼리 발생
```

**After 최적화**:
```
메시지 ID 리스트 추출
  ↓
IN 절로 읽음 상태 배치 조회
  ↓
1개 쿼리로 모든 읽음 상태 조회
```

**코드 예시**:
```java
// domain/chat/service/ChatMessageService.java
public List<ChatMessageDTO> getMessages(Long chatRoomId, Long userId) {
    // 메시지 조회
    List<ChatMessage> messages = chatMessageRepository.findByChatRoomIdOrderByCreatedAtDesc(chatRoomId);
    
    // 메시지 ID 리스트 추출
    List<Long> messageIds = messages.stream()
        .map(ChatMessage::getIdx)
        .collect(Collectors.toList());
    
    // 배치 조회: 읽음 상태
    Map<Long, Boolean> readStatusMap = chatReadStatusRepository
        .findByMessageIdxInAndUserId(messageIds, userId)
        .stream()
        .collect(Collectors.toMap(
            ChatReadStatus::getMessageIdx,
            ChatReadStatus::getIsRead
        ));
    
    // 매핑
    return messages.stream()
        .map(message -> {
            ChatMessageDTO dto = chatMessageConverter.toDTO(message);
            dto.setIsRead(readStatusMap.getOrDefault(message.getIdx(), false));
            return dto;
        })
        .collect(Collectors.toList());
}
```

#### 2. Redis 캐싱
**위치**: `domain/chat/service/ChatMessageService.java`

**전체 흐름**:
```
읽음 상태 조회
  ↓
Redis에서 먼저 조회 시도
  ↓
있으면: Redis에서 반환
없으면: DB에서 조회 후 Redis에 저장
```

**효과**:
- 조회 성능 향상
- DB 부하 감소

## 📝 핵심 정리

### WebSocket vs SSE
- **WebSocket**: 양방향, STOMP, 채팅
- **SSE**: 단방향, HTTP, 알림
- **선택 기준**: 통신 방향과 용도에 따라 선택

### WebSocket 채팅 시스템
- **설정**: WebSocketConfig에서 STOMP 엔드포인트 등록
- **메시지 전송**: SimpMessagingTemplate 사용
- **자동 연동**: 펫케어/모임 생성 시 채팅방 자동 생성
- **읽음 상태**: 배치 조회로 최적화

### SSE 알림 시스템
- **이중 저장소**: MySQL (영구) + Redis (최신 50개)
- **TTL**: 24시간
- **실시간 전송**: SseEmitter로 푸시
- **효과**: 빠른 조회, 실시간 알림

### 읽음 상태 최적화
- **배치 조회**: IN 절로 한 번에 조회
- **Redis 캐싱**: 자주 조회되는 데이터 캐싱
- **효과**: N+1 문제 해결, 성능 향상
