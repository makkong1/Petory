## 1. 프로젝트 개요 및 아키텍처

### Q1-1. Petory 프로젝트를 간단히 소개해주세요.
**답변 포인트:**
- 반려동물 통합 플랫폼: 커뮤니티, 펫케어 매칭, 실종 동물 찾기, 위치 기반 서비스, 오프라인 모임, 펫코인 결제, 채팅, 알림, 신고/제재
- 도메인 주도 설계(DDD)와 레이어드 아키텍처 적용
- 12개 도메인: User, Board, Care, Location, MissingPet, Meetup, Chat, Notification, Report, Statistics, File, Activity + Payment

### Q1-2. 프로젝트에서 도메인 주도 설계(DDD)를 어떻게 적용했나요?
**답변 포인트:**
- 도메인별 패키지 구조 (`domain/*/controller`, `service`, `entity`, `repository`, `exception`, `dto`)
- Repository + Jpa*Adapter + SpringDataJpa*Repository 패턴으로 데이터 액세스 추상화
- 도메인별 구체 예외 (BoardNotFoundException, UserNotFoundException 등)
- 명확한 도메인 경계와 책임 분리

### Q1-3. 레이어드 아키텍처의 각 레이어 역할을 설명해주세요.
**답변 포인트:**
- **Controller**: HTTP 요청/응답, REST API, WebSocket, SSE 엔드포인트
- **Service**: 비즈니스 로직, 트랜잭션 관리
- **Repository**: 도메인 인터페이스 (BoardRepository 등)
- **Jpa*Adapter**: Spring Data JPA 구현체 래핑
- **Entity**: 도메인 모델

### Q1-4. 프로젝트의 핵심 성과를 설명해주세요.
**답변 포인트:**
- **Board 도메인**: 쿼리 수 301개 → 3개 (99% 감소), 실행 시간 745ms → 30ms (24.83배 개선)
- **Location 도메인**: 조회 데이터 22,699개 → 1,026개 (95.5% 감소, ST_Distance_Sphere 반경 검색)
- **User 도메인**: 로그인 쿼리 21개 → 4개 (80.95% 감소, 채팅방 목록 배치 조회)
- **Payment 도메인**: 비관적 락으로 펫코인 에스크로 Race Condition 방지

## 🎯 핵심 키워드 정리

### 아키텍처
- 도메인 주도 설계 (DDD)
- Repository + Adapter 패턴
- 도메인별 구체 예외

### 성능 최적화
- N+1 문제 해결 (Fetch Join, 배치 조회)
- 스냅샷 패턴 (인기글, 통계)
- Redis 캐싱 (알림, 위치 서비스)

### 동시성 제어
- 원자적 증가 쿼리 (경고 횟수)
- 비관적 락 (펫코인 에스크로)
- 원자적 UPDATE (모임 참여 인원)

### 보안
- JWT (Access 15분 + Refresh 1일)
- OAuth2 (Google, Naver)
- BCrypt, RBAC

### 실시간 통신
- WebSocket (STOMP) - 채팅
- SSE - 알림
- 이중 저장소 (Redis + MySQL)
