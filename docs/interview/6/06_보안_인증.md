# 6. 보안 & 인증

## Q6-1. JWT 토큰 기반 인증 방식을 설명해주세요.

### 답변 포인트
- Access Token (15분) + Refresh Token (1일)
- Access Token 만료 시 Refresh Token으로 갱신
- JwtAuthenticationFilter에서 토큰 검증
- 토큰에 사용자 정보 포함 (idx, role 등)

### 상세 답변

#### 1. JWT 토큰 구조
**위치**: `util/JwtUtil.java`

**토큰 구성**:
- **Header**: 알고리즘 정보 (HS256)
- **Payload**: 사용자 정보 (idx, id, role 등)
- **Signature**: 서명 (비밀키로 생성)

**코드 예시**:
```java
// util/JwtUtil.java
public String generateToken(String id, Long idx, String role) {
    Map<String, Object> claims = new HashMap<>();
    claims.put("idx", idx);
    claims.put("role", role);
    
    return Jwts.builder()
        .setClaims(claims)
        .setSubject(id)
        .setIssuedAt(new Date())
        .setExpiration(new Date(System.currentTimeMillis() + jwtExpiration))
        .signWith(SignatureAlgorithm.HS256, secretKey)
        .compact();
}
```

#### 2. Access Token과 Refresh Token
**위치**: `domain/user/service/AuthService.java`

**Access Token**:
- 만료 시간: 15분 (86400000ms)
- 용도: API 요청 인증
- 저장 위치: 클라이언트 메모리 또는 로컬 스토리지

**Refresh Token**:
- 만료 시간: 1일
- 용도: Access Token 갱신
- 저장 위치: DB (Users.refreshToken)

**전체 흐름**:
```
로그인 성공
  ↓
Access Token + Refresh Token 생성
  ↓
Access Token: 클라이언트에 반환
Refresh Token: DB에 저장
  ↓
API 요청 시 Access Token 사용
  ↓
Access Token 만료 시
  ↓
Refresh Token으로 새 Access Token 발급
```

#### 3. JWT 필터에서 토큰 검증
**위치**: `filter/JwtAuthenticationFilter.java`
**메서드**: `doFilterInternal()`

**전체 흐름**:
```
요청 수신
  ↓
JwtAuthenticationFilter.doFilterInternal()
  ↓
Authorization 헤더에서 토큰 추출
  ↓
JwtUtil.validateToken() 검증
  ↓
토큰 만료 확인
  ↓
UserDetailsService.loadUserByUsername() 사용자 조회
  ↓
SecurityContext에 인증 정보 저장
  ↓
다음 필터로 전달
```

**코드 예시**:
```java
// filter/JwtAuthenticationFilter.java
@Override
protected void doFilterInternal(HttpServletRequest request,
        HttpServletResponse response,
        FilterChain filterChain) {
    
    String token = null;
    
    // 헤더에서 토큰 추출
    String authorizationHeader = request.getHeader("Authorization");
    if (authorizationHeader != null && authorizationHeader.startsWith("Bearer ")) {
        token = jwtUtil.extractTokenFromHeader(authorizationHeader);
    }
    
    // 쿼리 파라미터에서 토큰 추출 (SSE 등)
    if (token == null) {
        token = request.getParameter("token");
    }
    
    if (token != null && jwtUtil.validateToken(token)) {
        String id = jwtUtil.getIdFromToken(token);
        
        if (id != null && SecurityContextHolder.getContext().getAuthentication() == null) {
            UserDetails userDetails = userDetailsService.loadUserByUsername(id);
            
            if (jwtUtil.validateToken(token) && !jwtUtil.isTokenExpired(token)) {
                UsernamePasswordAuthenticationToken authToken = 
                    new UsernamePasswordAuthenticationToken(
                        userDetails, null, userDetails.getAuthorities());
                
                SecurityContextHolder.getContext().setAuthentication(authToken);
            }
        }
    }
    
    filterChain.doFilter(request, response);
}
```

#### 4. 토큰 갱신
**위치**: `domain/user/controller/AuthController.java`

**전체 흐름**:
```
Access Token 만료
  ↓
Refresh Token으로 갱신 요청
  ↓
Refresh Token 검증
  ↓
DB에서 사용자 조회 (Refresh Token으로)
  ↓
새 Access Token 발급
  ↓
클라이언트에 반환
```

## Q6-2. OAuth2 소셜 로그인을 어떻게 구현했나요?

### 답변 포인트
- Google, Naver 지원
- Spring Security OAuth2 Client 활용
- Provider별 OAuth2UserService에서 표준화
- 기존 사용자 자동 연결, 신규 사용자 자동 회원가입

### 상세 답변

#### 1. OAuth2 설정
**위치**: `application.properties`

**Google 설정**:
```properties
spring.security.oauth2.client.registration.google.client-id=...
spring.security.oauth2.client.registration.google.client-secret=...
spring.security.oauth2.client.registration.google.scope=profile,email
spring.security.oauth2.client.registration.google.redirect-uri={baseUrl}/login/oauth2/code/{registrationId}
```

**Naver 설정**:
```properties
spring.security.oauth2.client.registration.naver.client-id=...
spring.security.oauth2.client.registration.naver.client-secret=...
spring.security.oauth2.client.registration.naver.scope=name,email
spring.security.oauth2.client.provider.naver.authorization-uri=https://nid.naver.com/oauth2.0/authorize
spring.security.oauth2.client.provider.naver.token-uri=https://nid.naver.com/oauth2.0/token
spring.security.oauth2.client.provider.naver.user-info-uri=https://openapi.naver.com/v1/nid/me
```

#### 2. Provider별 커스텀 클라이언트
**위치**: `domain/user/service/ConditionalOAuth2TokenResponseClient.java`, `OAuth2UserProviderRouter`

**전체 흐름**:
```
OAuth2 인증 요청
  ↓
ConditionalOAuth2TokenResponseClient.getTokenResponse()
  ↓
Provider 확인 (naver / google)
  ↓
Naver: NaverOAuth2TokenResponseClient 사용
Google: RestClientAuthorizationCodeTokenResponseClient 사용
  ↓
토큰 발급
```

**코드 예시**:
```java
// domain/user/service/ConditionalOAuth2TokenResponseClient.java
@Override
public OAuth2AccessTokenResponse getTokenResponse(
        OAuth2AuthorizationCodeGrantRequest authorizationCodeGrantRequest) {
    
    String registrationId = authorizationCodeGrantRequest.getClientRegistration().getRegistrationId();
    
    // Naver인 경우 커스텀 클라이언트 사용
    if ("naver".equalsIgnoreCase(registrationId)) {
        return naverClient.getTokenResponse(authorizationCodeGrantRequest);
    }
    
    // Google 등 다른 provider는 기본 클라이언트 사용
    return getDefaultClient().getTokenResponse(authorizationCodeGrantRequest);
}
```

#### 3. OAuth2UserService
**위치**: `domain/user/service/OAuth2UserProviderRouter` (userInfoEndpoint.userService에 등록)

**전체 흐름**:
```
소셜 로그인 콜백
  ↓
OAuth2UserService.loadUser()
  ↓
Provider별 사용자 정보 추출
  ↓
기존 사용자 확인 (email 또는 providerId)
  ↓
있으면: 기존 계정 반환
없으면: 새 계정 생성 후 반환
```

**코드 예시**:
```java
// domain/user/service/OAuth2UserService.java
@Override
public OAuth2User loadUser(OAuth2UserRequest userRequest) throws OAuth2AuthenticationException {
    OAuth2User oAuth2User = super.loadUser(userRequest);
    
    String registrationId = userRequest.getClientRegistration().getRegistrationId();
    
    // Provider별 사용자 정보 추출
    Map<String, Object> attributes = oAuth2User.getAttributes();
    
    // 기존 사용자 확인
    Users user = processOAuth2User(userRequest, oAuth2User);
    
    return new DefaultOAuth2User(
        user.getAuthorities(),
        attributes,
        "id"
    );
}
```

#### 4. 자동 회원가입
**위치**: `domain/user/service/OAuth2UserService.java`
**메서드**: `processOAuth2User()`

**전체 흐름**:
```
소셜 계정 정보 추출
  ↓
SocialUser 조회 시도
  ↓
없으면: 새 계정 생성
  - Users 엔티티 생성
  - SocialUser 엔티티 생성
  - 이메일 인증 자동 완료
  ↓
있으면: 기존 계정 반환
```

## Q6-3. 이메일 인증 시스템을 설명해주세요.

### 답변 포인트
- 단일 이메일 인증 시스템
- Redis에 임시 인증 정보 저장
- 이메일 인증 완료 후 서비스 이용 가능
- 소셜 로그인 시 자동 인증 완료

### 상세 답변

#### 1. 이메일 인증 토큰 생성
**위치**: `domain/user/service/EmailVerificationService.java`
**메서드**: `sendVerificationEmail()`

**전체 흐름**:
```
회원가입 또는 이메일 변경
  ↓
인증 토큰 생성 (UUID)
  ↓
Redis에 저장 (TTL: 24시간)
  ↓
이메일 발송
  ↓
사용자가 링크 클릭
  ↓
토큰 검증 및 인증 완료
```

**코드 예시**:
```java
// domain/user/service/EmailVerificationService.java
public void sendVerificationEmail(Long userId, EmailVerificationPurpose purpose) {
    String token = UUID.randomUUID().toString();
    
    // Redis에 저장
    String key = "email_verification:" + token;
    EmailVerificationData data = EmailVerificationData.builder()
        .userId(userId)
        .purpose(purpose)
        .build();
    
    redisTemplate.opsForValue().set(key, data, 24, TimeUnit.HOURS);
    
    // 이메일 발송
    emailService.sendVerificationEmail(user.getEmail(), token, purpose);
}
```

#### 2. 이메일 인증 확인
**위치**: `domain/user/service/EmailVerificationService.java`
**메서드**: `verifyEmail()`

**전체 흐름**:
```
인증 링크 클릭
  ↓
토큰 추출
  ↓
Redis에서 인증 정보 조회
  ↓
토큰 유효성 확인
  ↓
사용자 이메일 인증 상태 업데이트
  ↓
Redis에서 토큰 삭제
```

**코드 예시**:
```java
// domain/user/service/EmailVerificationService.java
public void verifyEmail(String token) {
    String key = "email_verification:" + token;
    EmailVerificationData data = (EmailVerificationData) redisTemplate.opsForValue().get(key);
    
    if (data == null) {
        throw new IllegalArgumentException("유효하지 않은 인증 토큰입니다.");
    }
    
    Users user = usersRepository.findById(data.getUserId())
        .orElseThrow(() -> new IllegalArgumentException("사용자를 찾을 수 없습니다."));
    
    user.setEmailVerified(true);
    usersRepository.save(user);
    
    redisTemplate.delete(key);
}
```

#### 3. 인증 필요 기능 제한
**위치**: `domain/user/exception/EmailVerificationRequiredException.java`

**사용 예시**:
```java
// domain/board/service/BoardService.java
if (user.getEmailVerified() == null || !user.getEmailVerified()) {
    throw new EmailVerificationRequiredException(
        "게시글 작성을 위해 이메일 인증이 필요합니다.",
        EmailVerificationPurpose.BOARD_CREATE);
}
```

## Q6-4. 비밀번호 암호화 방식을 설명해주세요.

### 답변 포인트
- BCryptPasswordEncoder 사용
- Salt 자동 생성
- 단방향 해시 함수

### 상세 답변

#### 1. BCrypt 설정
**위치**: `global/security/SecurityConfig.java`

**코드 예시**:
```java
@Bean
public PasswordEncoder passwordEncoder() {
    return new BCryptPasswordEncoder();
}
```

#### 2. 비밀번호 암호화
**위치**: `domain/user/service/AuthService.java`

**전체 흐름**:
```
회원가입 시 비밀번호 입력
  ↓
BCryptPasswordEncoder.encode()
  ↓
Salt 자동 생성 + 해시
  ↓
암호화된 비밀번호 DB 저장
```

**코드 예시**:
```java
// domain/user/service/AuthService.java
public Users register(String username, String password, String email) {
    String encodedPassword = passwordEncoder.encode(password);
    
    Users user = Users.builder()
        .username(username)
        .password(encodedPassword)
        .email(email)
        .build();
    
    return usersRepository.save(user);
}
```

#### 3. 비밀번호 검증
**위치**: `domain/user/service/AuthService.java`

**코드 예시**:
```java
// domain/user/service/AuthService.java
public boolean authenticate(String username, String password) {
    Users user = usersRepository.findByUsername(username)
        .orElseThrow(() -> new IllegalArgumentException("사용자를 찾을 수 없습니다."));
    
    return passwordEncoder.matches(password, user.getPassword());
}
```

**BCrypt 특징**:
- Salt 자동 생성 (각 비밀번호마다 다른 Salt)
- 단방향 해시 (복호화 불가능)
- 느린 해시 함수 (무차별 대입 공격 방지)

## Q6-5. 역할 기반 접근 제어(RBAC)를 어떻게 구현했나요?

### 답변 포인트
- Role: USER, ADMIN, SERVICE_PROVIDER, MASTER
- @PreAuthorize 어노테이션으로 권한 체크
- SecurityConfig에서 경로별 권한 설정

### 상세 답변

#### 1. Role 정의
**위치**: `domain/user/entity/Users.java`

**Role 종류**:
- `USER`: 일반 사용자
- `ADMIN`: 관리자
- `SERVICE_PROVIDER`: 서비스 제공자
- `MASTER`: 최고 관리자

**코드 예시**:
```java
// domain/user/entity/Users.java
@Enumerated(EnumType.STRING)
private Role role;
```

#### 2. SecurityConfig 설정
**위치**: `global/security/SecurityConfig.java`

**전체 흐름**:
```
요청 수신
  ↓
SecurityConfig에서 경로별 권한 체크
  ↓
공개 API: permitAll()
보호된 API: authenticated()
관리자 API: hasRole('ADMIN')
  ↓
권한 없으면 403 Forbidden
```

**코드 예시**:
```java
// global/security/SecurityConfig.java
.authorizeHttpRequests(auth -> auth
    .requestMatchers("/api/auth/**").permitAll()
    .requestMatchers("/api/master/**").hasRole("MASTER")
    .requestMatchers("/api/admin/**").hasAnyRole("ADMIN", "MASTER")
    .requestMatchers("/api/**").authenticated()
    .anyRequest().permitAll()
)
```

#### 3. 메서드 레벨 권한 체크
**위치**: `PetoryApplication.java` (`@EnableMethodSecurity`)

**사용 예시**:
```java
// domain/admin/controller/AdminUserController.java
@PreAuthorize("hasRole('ADMIN')")
@GetMapping("/users")
public List<UserDTO> getAllUsers() {
    return userService.getAllUsers();
}
```

**활성화**:
```java
// PetoryApplication.java
@EnableMethodSecurity(prePostEnabled = true)
public class PetoryApplication {
    // ...
}
```

## Q6-6. SQL Injection과 XSS 공격을 어떻게 방지했나요?

### 답변 포인트
- **SQL Injection**: JPA 사용으로 자동 방지 (PreparedStatement)
- **XSS**: DTO 변환 시 이스케이프 처리
- 경로 정규화로 Directory Traversal 공격 방지

### 상세 답변

#### 1. SQL Injection 방지
**방법**: JPA 사용

**전체 흐름**:
```
JPA 쿼리 작성
  ↓
JPQL 또는 쿼리 메서드
  ↓
Hibernate가 PreparedStatement로 변환
  ↓
파라미터 바인딩
  ↓
SQL Injection 자동 방지
```

**코드 예시**:
```java
// 안전한 방법: 파라미터 바인딩
@Query("SELECT b FROM Board b WHERE b.title = :title")
List<Board> findByTitle(@Param("title") String title);

// 위험한 방법: 문자열 연결 (사용하지 않음)
// @Query("SELECT b FROM Board b WHERE b.title = '" + title + "'")
```

#### 2. XSS 방지
**방법**: DTO 변환 시 이스케이프 처리

**위치**: `domain/board/converter/BoardConverter.java`

**코드 예시**:
```java
// domain/board/converter/BoardConverter.java
public BoardDTO toDTO(Board board) {
    return BoardDTO.builder()
        .title(escapeHtml(board.getTitle()))
        .content(escapeHtml(board.getContent()))
        .build();
}

private String escapeHtml(String input) {
    if (input == null) return null;
    return input
        .replace("&", "&amp;")
        .replace("<", "&lt;")
        .replace(">", "&gt;")
        .replace("\"", "&quot;")
        .replace("'", "&#x27;");
}
```

#### 3. Directory Traversal 방지
**위치**: `domain/file/service/FileService.java`

**방법**: 경로 정규화 및 검증

**코드 예시**:
```java
// domain/file/service/FileService.java
public String normalizePath(String path) {
    Path normalized = Paths.get(path).normalize();
    
    // 상위 디렉토리 접근 방지
    if (normalized.startsWith("..")) {
        throw new IllegalArgumentException("Invalid path");
    }
    
    return normalized.toString();
}
```

## 📝 핵심 정리

### JWT 인증
- **Access Token**: 15분 만료
- **Refresh Token**: 1일 만료, DB 저장
- **필터**: JwtAuthenticationFilter에서 검증
- **효과**: Stateless 인증

### OAuth2 소셜 로그인
- **Provider**: Google, Naver
- **커스텀 클라이언트**: NaverOAuth2TokenResponseClient
- **자동 회원가입**: 기존 계정 확인 후 없으면 생성
- **효과**: 사용자 편의성 향상

### 이메일 인증
- **토큰 저장**: Redis (TTL 24시간)
- **자동 인증**: 소셜 로그인 시
- **기능 제한**: 인증 전 일부 기능 제한
- **효과**: 스팸 계정 방지

### 비밀번호 암호화
- **방식**: BCryptPasswordEncoder
- **Salt**: 자동 생성
- **특징**: 단방향 해시, 느린 해시 함수
- **효과**: 무차별 대입 공격 방지

### RBAC
- **Role**: USER, ADMIN, SERVICE_PROVIDER, MASTER
- **경로별 권한**: SecurityConfig
- **메서드 레벨**: @PreAuthorize
- **효과**: 세밀한 권한 제어

### 보안 공격 방지
- **SQL Injection**: JPA PreparedStatement
- **XSS**: HTML 이스케이프 처리
- **Directory Traversal**: 경로 정규화
- **효과**: 다양한 공격 방지
