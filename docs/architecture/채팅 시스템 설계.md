# 채팅 시스템 설계

> 현재 코드 기준 재작성. 구형 `requester_idx`/`provider_idx` 분리 구조, `MessageReadStatus` 테이블, 루프 기반 unreadCount 업데이트는 **모두 제거됨**.

---

## 1. 채팅방 타입

| `ConversationType` | 설명 | 생성 트리거 |
|--------------------|------|-------------|
| `DIRECT` | 1:1 일반 채팅 | 사용자가 직접 생성 (`POST /api/chat/conversations/direct`) |
| `GROUP` | 그룹 채팅 | 수동 생성 |
| `CARE_REQUEST` | 펫케어 요청 채팅 | 거래 확정 흐름 (`POST /api/chat/conversations/care-request`) |
| `MISSING_PET` | 실종제보 채팅 | 연관 실종 제보와 함께 생성 |
| `MEETUP` | 산책·모임 채팅 | 모임 생성 시 자동 생성 (이벤트 기반) |
| `ADMIN_SUPPORT` | 관리자 지원 채팅 | 수동 |

---

## 2. 핵심 엔티티 구조

```
Conversation
  ├── ConversationType conversationType
  ├── RelatedType relatedType       (CARE_REQUEST, CARE_APPLICATION, MISSING_PET_BOARD, MEETUP, USER)
  ├── Long relatedIdx
  └── List<ConversationParticipant> participants

ConversationParticipant
  ├── Conversation conversation
  ├── User user
  ├── ParticipantStatus status      (ACTIVE, LEFT, KICKED, MUTED)
  └── int unreadCount               ← 참여자별 안 읽은 메시지 수 (원자적 UPDATE)

ChatMessage
  ├── Conversation conversation
  ├── User sender
  ├── String content
  └── MessageType type              (TEXT, IMAGE, FILE, SYSTEM, NOTICE)
```

`MessageReadStatus` 테이블은 **존재하지 않음**. 읽음 상태는 `ConversationParticipant.unreadCount`로 관리한다.

---

## 3. 읽지 않은 메시지 수 (원자적 UPDATE)

메시지 전송 시 발신자를 제외한 모든 `ACTIVE` 참여자의 `unreadCount`를 원자적으로 증가:

```java
@Modifying
@Query("UPDATE ConversationParticipant p SET p.unreadCount = p.unreadCount + 1 " +
       "WHERE p.conversation.idx = :conversationIdx " +
       "AND p.user.idx != :senderUserId " +
       "AND p.status = 'ACTIVE'")
void incrementUnreadCount(Long conversationIdx, Long senderUserId);
```

읽음 처리(`markAsRead`) 시:

```java
@Modifying
@Query("UPDATE ConversationParticipant p SET p.unreadCount = 0 " +
       "WHERE p.conversation.idx = :conversationIdx AND p.user.idx = :userId")
void resetUnreadCount(Long conversationIdx, Long userId);
```

루프 기반 읽음 처리는 제거됨. DB 레벨 원자적 UPDATE로 Lost Update 방지.

## 4. STOMP WebSocket

### 연결 엔드포인트

```
ws://host/ws?token=<JWT>     (또는 /chat)
SockJS 폴백 지원
```

인증: `WebSocketAuthenticationInterceptor` → HandshakeHandler → Principal 주입  
채널 인터셉터: `WebSocketAuthChannelInterceptor` (메시지 레벨 토큰 검증)

### 메시지 브로커 프리픽스

| 방향 | 프리픽스 | 설명 |
|------|----------|------|
| 클라 → 서버 전송 | `/app` | `@MessageMapping` 대상 |
| 서버 → 클라 구독 | `/topic`, `/queue`, `/user` | SimpleBroker |

### STOMP 엔드포인트

| 클라이언트 → 서버 | 설명 |
|------------------|------|
| `SEND /app/chat.send` | 메시지 전송 |
| `SEND /app/chat.read` | 읽음 처리 |
| `SEND /app/chat.typing` | 타이핑 인디케이터 |

| 서버 → 클라이언트 구독 | 설명 |
|------------------------|------|
| `/topic/conversation/{conversationIdx}` | 채팅방 메시지 수신 |
| `/user/queue/messages` | 개인 알림 |

## 5. 채팅방 자동 생성 흐름

### 5.1 펫케어 채팅방 (CARE_REQUEST)

```
사용자가 「지원 승인」 또는 「채팅하기」 클릭
  → POST /api/chat/conversations/care-request?careApplicationIdx=N
  → ConversationService.createCareRequestConversation()
      · conversationType = CARE_REQUEST
      · relatedType = CARE_APPLICATION, relatedIdx = careApplicationIdx
      · 보호자 + 시터를 ConversationParticipant에 추가
      · 이미 존재하면 기존 방 반환
```

`POST /api/chat/conversations` 일반 생성 시에는 `conversationType: DIRECT`, `relatedType: CARE_REQUEST`로도 생성 가능 (프론트 상황에 따라 선택).

### 5.2 모임 채팅방 (MEETUP) — 이벤트 기반

```
MeetupService.createMeetup()
  → TransactionSynchronizationManager.registerSynchronization(afterCommit)
      → ApplicationEventPublisher.publishEvent(MeetupCreatedEvent)

MeetupChatRoomEventListener (@Async + @Transactional(REQUIRES_NEW))
  → ConversationService.createMeetupChatRoom()
      · conversationType = MEETUP
      · relatedType = MEETUP, relatedIdx = meetupIdx
      · 주최자를 ConversationParticipant에 추가
```

**`afterCommit` 후 이벤트 발행**: 모임 생성 트랜잭션 커밋 완료 후 발행하여 채팅방이 반드시 존재하는 상태를 보장.  
**`REQUIRES_NEW`**: 채팅방 생성 트랜잭션이 모임 생성 트랜잭션과 분리 — 채팅방 생성 실패 시 모임이 롤백되지 않음.

모임 참여 시 채팅 자동 가입:

```
POST /api/meetups/{meetupIdx}/join
  → ConversationParticipant에 참여자 추가 (joinMeetupChat)
```

## 7. 보안 정책

- **채팅방 접근**: `ConversationParticipant`에 해당 사용자가 `ACTIVE` 상태로 존재해야 메시지 조회·전송 가능.
- **WebSocket 인증**: 연결(Handshake) 시 JWT 검증 + 채널 인터셉터에서 재검증.
- **메시지 삭제**: 발신자 본인만 가능.

---

## 8. 거래 확정 흐름 (`confirmCareDeal`)

```
POST /api/chat/conversations/{idx}/confirm-deal
  → ConversationService.confirmCareDeal()
      · 양쪽(보호자·시터) 모두 확정 표시
      · 둘 다 확정 시:
          - CareApplication 상태 → ACCEPTED
          - CareRequest 상태 → IN_PROGRESS
          - PetCoinEscrowService.createEscrow() (실패 시 현재 로그만 남기고 진행)
```

> **주의**: `createEscrow` 실패 시 예외를 삼켜 로그만 남기는 현행 구현은 운영 정책에 따라 롤백 여부 재검토 필요.
