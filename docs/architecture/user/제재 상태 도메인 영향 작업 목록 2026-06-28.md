# 제재 상태 도메인 영향 작업 목록 (2026-06-28)

## 1. 목적

이 문서는 코드 수정 전에, 기존 도메인 문서만 기준으로 `SUSPENDED` / `BANNED` 사용자가 각 도메인에 미치는 영향을 정리한다.

현재 1차 작업 대상은 다음 두 축이다.

1. 문제상황 문서화: [제재 상태 인증 우회 문제 정리](../../troubleshooting/users/sanction-auth-gap-2026-06-28.md)
2. 도메인별 추가 로직 설계: 이 문서

## 2. 판단 기준

제재 상태는 로그인 차단만으로 끝나지 않는다. 사용자가 이미 도메인 객체의 작성자, 참가자, 제공자, 채팅 참여자인 경우가 있기 때문이다.

기본 판단 기준:

| 상태 | 기본 정책 |
|---|---|
| `ACTIVE` | 기존 기능 사용 가능 |
| `SUSPENDED` | 신규 생성/참가/지원/채팅 전송 차단. 기존 진행 건은 도메인별 정책 필요 |
| `BANNED` | 신규 행동 차단. 기존 노출/참여/진행 건 정리 필요 |

## 3. 우선 점검 도메인

| 우선순위 | 도메인 | 이유 |
|---|---|---|
| 0 | User/Auth/Report | 제재 상태가 인증과 신고 처리에서 실제로 적용되어야 다른 도메인 정책이 의미 있음 |
| 1 | Care | 요청자, 제공자, 채팅, 결제/에스크로가 얽혀 있어 제재 영향이 가장 큼 |
| 2 | Meetup | 모임 생성자/참가자/그룹채팅이 있고, 차단 시 모임 취소 정책이 필요 |
| 3 | Chat | Care와 Meetup의 사용자 간 접점. 제재 사용자 안내/전송 차단/방 상태 처리가 필요 |
| 4 | Payment | Care 진행 중 제재 시 지급/환불/보류 정책이 필요. 자동 처리보다 관리자 검토가 안전 |
| 5 | Board/MissingPet/Comment/Review | 콘텐츠 노출과 작성자 비활성화 정책이 필요하지만, 1차 범위는 Care/Meetup 이후 |
| 6 | Notification/Statistics | 제재 후 알림 발송 대상과 통계 집계 기준 조정 여부 확인 |

## 4. 1차 필수 전제 작업

Care/Meetup 도메인 로직을 넣기 전에 아래가 먼저 정리되어야 한다.

| 작업 | 필요 이유 |
|---|---|
| refresh token 제재 검사 | 정지/차단 사용자가 새 access token을 계속 받으면 도메인 차단이 무력화됨 |
| JWT/WebSocket 인증 제재 검사 | 기존 access token과 채팅 연결에서 제재 상태가 반영되어야 함 |
| 정지/차단 시 refresh token 제거 | 제재 즉시성을 높이기 위한 최소 조치 |
| 신고 target별 제재 대상 user resolve | 콘텐츠 ID를 사용자 ID로 오인해 잘못 제재하는 문제 방지 |
| 제재 도메인 이벤트 또는 공통 처리 지점 정의 | 사용자가 제재될 때 Care/Meetup/Chat 후속 처리를 안정적으로 호출하기 위함 |

### 5.1 요청자가 제재된 경우

사용자 말 기준 정책: “사용자는 글 자체가 안 보이게”

작업:

| 상황 | 필요한 동작 |
|---|---|
| 요청자가 `SUSPENDED` | 신규 케어 요청 생성 차단 |
| 요청자가 `BANNED` | 신규 케어 요청 생성 차단 |
| 요청자가 제재된 기존 `OPEN` 요청 | 목록/검색/지도/추천/상세에서 비노출 또는 관리자 전용 상태로 전환 |
| 요청자가 제재된 기존 `IN_PROGRESS` 요청 | 자동 완료/지급 금지. 관리자 검토 상태 또는 취소/환불 정책 필요 |
| 요청자가 제재된 기존 채팅방 | 상대 제공자에게 안내 메시지 표시, 신규 거래 확정 차단 |

확인할 API/흐름:

- 케어 목록, 검색, nearby, 상세
- 케어 요청 생성/수정/삭제
- 케어 상태 변경
- 케어 자동 완료 스케줄러
- 채팅 거래 확정
- 에스크로 생성/환불/지급

### 5.2 제공자가 제재된 경우

사용자 말 기준 정책: “제공자는 채팅에서 ‘신고당한 유저입니다’처럼 안내하고 안 보이게”

작업:

| 상황 | 필요한 동작 |
|---|---|
| 제공자가 `SUSPENDED` | 신규 케어 댓글/지원/거래 확정 차단 |
| 제공자가 `BANNED` | 신규 케어 댓글/지원/거래 확정 차단 |
| 제공자가 제재된 채팅방 | 요청자 화면에 “신고/제재된 사용자” 안내 메시지 표시 |
| 제공자가 제재된 기존 `OPEN` 매칭 전 대화 | 거래 확정 차단, 필요 시 채팅방 닫기 또는 읽기 전용 |
| 제공자가 제재된 기존 `IN_PROGRESS` 케어 | 지급 자동 진행 금지, 관리자 검토/재배정/환불 정책 필요 |

주의:

- 현재 Care 매칭은 Chat 도메인의 `confirmCareDeal()`과 강하게 연결되어 있다.
- 따라서 Care만 막아서는 부족하고 Chat 메시지 전송, 거래 확정, 참가자 노출 정책을 같이 수정해야 한다.

### 5.3 Care 테스트 후보

| 테스트 | 기대 |
|---|---|
| 제재 요청자의 `OPEN` 케어 요청이 목록/검색/지도에서 제외된다 | 비노출 |
| 제재 요청자의 케어 상세 접근 정책이 적용된다 | 일반 사용자 비노출 또는 안내 응답 |
| 제재 요청자는 케어 요청을 새로 만들 수 없다 | 403 |
| 제재 제공자는 케어 댓글/지원/거래 확정을 할 수 없다 | 403 |
| 제재 제공자가 포함된 채팅방은 상대방에게 안내 상태를 내려준다 | 안내 표시 가능 DTO |
| 진행 중 케어의 요청자/제공자가 제재되면 자동 완료/지급이 진행되지 않는다 | 관리자 검토/보류 |

사용자 말 기준 정책: “모임은 당연히 취소”

### 6.1 주최자가 제재된 경우

작업:

| 상황 | 필요한 동작 |
|---|---|
| 주최자가 `SUSPENDED` | 신규 모임 생성 차단 |
| 주최자가 `BANNED` | 신규 모임 생성 차단 |
| 주최자가 제재된 기존 모집 중 모임 | 모임 취소 또는 soft delete. 목록/검색/추천/nearby 비노출 |
| 주최자가 제재된 기존 `CLOSED` 모임 | 취소 처리 및 참가자 안내 필요 |
| 주최자가 제재된 모임 채팅방 | 신규 메시지 차단, 시스템 안내 메시지 또는 방 종료 |

### 6.2 참가자가 제재된 경우

작업:

| 상황 | 필요한 동작 |
|---|---|
| 제재 사용자의 모임 참가 | 차단 |
| 제재 사용자의 기존 참가 row | 참가 취소 처리 또는 참가자 목록 비노출 |
| 제재 사용자의 모임 채팅 참여 | 채팅방에서 나가기 또는 읽기 전용/안내 처리 |
| 제재 사용자의 히스토리 | 본인 히스토리 보존 여부와 타인 노출 여부 분리 |

### 6.3 Meetup 테스트 후보

| 테스트 | 기대 |
|---|---|
| 제재 사용자는 모임을 생성할 수 없다 | 403 |
| 제재 사용자는 모임에 참가할 수 없다 | 403 |
| 주최자가 제재되면 모집 중 모임이 취소/비노출된다 | 목록/검색/추천 제외 |
| 주최자가 제재되면 모임 채팅방이 닫히거나 안내 상태가 된다 | 메시지 전송 차단 |
| 참가자가 제재되면 참가자 목록/채팅 참여 정책이 적용된다 | 비노출 또는 퇴장 |

## 7. Chat 도메인 공통 작업

Care와 Meetup 모두 Chat을 거치므로 공통 정책이 필요하다.

작업:

| 상황 | 필요한 동작 |
|---|---|
| 제재 사용자의 REST 메시지 전송 | 차단 |
| 제재 사용자의 STOMP 메시지 전송 | 차단 |
| 제재 사용자가 포함된 채팅방 목록/상세 | 상대방에게 “신고/제재된 사용자” 안내를 내려줄 DTO 필요 |
| 진행 중 케어 채팅 상대가 제재됨 | 거래 확정 버튼 비활성화 및 안내 |
| 모임 채팅 참여자가 제재됨 | 메시지 전송 차단, 참가자 표시 정책 적용 |

주의:

- 상대방의 과거 메시지를 즉시 삭제할지, “제재된 사용자” placeholder로 가릴지는 프론트/백엔드 DTO 정책이 필요하다.
- 법적/운영 추적을 위해 DB 메시지를 삭제하기보다는 사용자 화면에서 마스킹하는 방식이 안전하다.

## 9. 아직 결정이 필요한 정책

| 정책 | 선택지 |
|---|---|
| 제재 요청자의 기존 `OPEN` 케어 요청 | soft delete / 별도 숨김 상태 / 목록만 비노출 |
| 진행 중 케어의 요청자 제재 | 환불 / 관리자 검토 / 일정 기간 유예 |
| 진행 중 케어의 제공자 제재 | 지급 보류 / 재배정 / 관리자 검토 |
| 제재 사용자의 과거 채팅 메시지 | 유지 / 프론트 마스킹 / 백엔드 DTO 마스킹 |
| 모임 주최자 제재 | 즉시 취소 / 관리자 검토 후 취소 |
| 모임 참가자 제재 | 참가 취소 / 참가자 목록 마스킹 / 채팅만 차단 |
