# 도메인별 트러블슈팅 체크리스트

## 📋 개요

각 도메인에서 발생할 수 있는 주요 트러블슈팅 포인트와 개선 방안을 정리합니다. **인덱스, 트랜잭션, 락, 자료구조/알고리즘** 적용이 필요한 부분을 도메인별로 정리했습니다.

## 1. 🎯 Meetup (모임) 도메인 (했음)

### ⚠️ 가장 심각한 문제: Race Condition (동시성 문제)

**위치**: `MeetupService.joinMeetup()`, `MeetupService.cancelMeetupParticipation()`

**문제 상황**:
```java
// MeetupService.java:277-279
if (meetup.getCurrentParticipants() >= meetup.getMaxParticipants()) {
    throw new RuntimeException("모임 인원이 가득 찼습니다.");
}
// ... 
meetup.setCurrentParticipants(meetup.getCurrentParticipants() + 1); // ⚠️ Race Condition!
```

**시나리오**:
- 동시에 2명이 모임에 참가 시도
- 둘 다 `currentParticipants < maxParticipants` 체크 통과
- 둘 다 참가 성공 → 인원 초과 발생!

**해결 방안**:

#### 1.1 Pessimistic Lock 적용 (즉시 적용 가능)
```java
@Transactional
public MeetupParticipantsDTO joinMeetup(Long meetupIdx, String userId) {
    // Pessimistic Lock으로 동시 접근 방지
    Meetup meetup = meetupRepository.findById(meetupIdx)
        .orElseThrow(() -> new RuntimeException("모임을 찾을 수 없습니다."));
    
    // Lock 적용
    Meetup lockedMeetup = meetupRepository.findByIdWithLock(meetupIdx)
        .orElseThrow(() -> new RuntimeException("모임을 찾을 수 없습니다."));
    
    // 인원 체크 및 증가를 원자적으로 처리
    if (lockedMeetup.getCurrentParticipants() >= lockedMeetup.getMaxParticipants()) {
        throw new RuntimeException("모임 인원이 가득 찼습니다.");
    }
    
    lockedMeetup.setCurrentParticipants(lockedMeetup.getCurrentParticipants() + 1);
    meetupRepository.save(lockedMeetup);
    
    // ... 참가자 추가 로직
}
```

**Repository에 추가**:
```java
@Lock(LockModeType.PESSIMISTIC_WRITE)
@Query("SELECT m FROM Meetup m WHERE m.idx = :idx")
Optional<Meetup> findByIdWithLock(@Param("idx") Long idx);
```

#### 1.2 Optimistic Lock 적용 (더 나은 성능)
```java
@Entity
public class Meetup {
    @Version
    private Long version; // Optimistic Lock용 버전 필드
    
    // ...
}
```

#### 1.3 DB 레벨 제약조건 추가
```sql (적용)
ALTER TABLE meetup 
ADD CONSTRAINT chk_participants 
CHECK (current_participants <= max_participants);
```

### 📊 인덱스 최적화

**현재 인덱스 상태** (✅ 대부분 이미 적용됨):
```sql
-- meetup 테이블
✅ idx_meetup_date_status (date, status) - 이미 존재
✅ idx_meetup_location (latitude, longitude) - 이미 존재
✅ organizer_idx (organizer_idx) - 이미 존재
⚠️ idx_meetup_date (date) - 단일 인덱스, 복합 인덱스와 중복 가능성
⚠️ idx_meetup_status (status) - 단일 인덱스, 복합 인덱스와 중복 가능성

-- meetup_participants 테이블
✅ PRIMARY KEY (meetup_idx, user_idx) - 이미 존재 (복합 기본키)
✅ user_idx (user_idx) - 이미 존재
```

**분석 결과**:
- ✅ 필요한 인덱스는 모두 존재함
- ⚠️ `idx_meetup_date`, `idx_meetup_status` 단일 인덱스는 `idx_meetup_date_status` 복합 인덱스와 중복될 수 있음
  - 복합 인덱스가 있으면 단일 인덱스는 불필요할 수 있음 (쿼리 패턴에 따라 다름)
  - `WHERE date = ?` 쿼리는 복합 인덱스로도 처리 가능
  - `WHERE status = ?` 쿼리는 복합 인덱스의 두 번째 컬럼이라 효과가 낮을 수 있음

**권장 사항**:
- 현재 상태로도 충분함
- 성능 모니터링 후 필요시 단일 인덱스 제거 고려 (복합 인덱스만 유지)

**현재 문제**: `getNearbyMeetups()` 메서드가 전체 모임을 조회 후 Java에서 필터링 (line 158)
- **개선**: DB 레벨에서 거리 계산 및 필터링 (Haversine 공식)

### 🔄 트랜잭션 개선 ✅ 적용 완료

**문제**: 채팅방 생성 실패 시 모임 생성까지 롤백하는 문제

**설계 원칙**: **파생 도메인은 실패해도 핵심 도메인을 롤백하면 안 된다.**

**해결**: 이벤트 기반 아키텍처로 핵심 도메인과 파생 도메인 분리

**적용 코드**:

#### 1. 모임 생성 (핵심 도메인 - 단독 트랜잭션)
```java
@Transactional
public MeetupDTO createMeetup(...) {
    Meetup savedMeetup = meetupRepository.save(meetup);
    
    // 이벤트 발행 (트랜잭션 커밋 후 비동기 처리)
    eventPublisher.publishEvent(new MeetupCreatedEvent(
            this,
            savedMeetup.getIdx(),
            organizer.getIdx(),
            savedMeetup.getTitle()
    ));
    
    return converter.toDTO(savedMeetup);
}
```

#### 2. 채팅방 생성 (파생 도메인 - 별도 트랜잭션)
```java
@EventListener
@Async
@Transactional(propagation = Propagation.REQUIRES_NEW)
public void handleMeetupCreated(MeetupCreatedEvent event) {
    try {
        conversationService.createConversation(...);
        conversationService.setParticipantRole(...);
    } catch (Exception e) {
        // 채팅방 생성 실패해도 모임은 이미 생성됨 (롤백되지 않음)
        log.error("채팅방 생성 실패: meetupIdx={}", event.getMeetupIdx(), e);
    }
}
```

**효과**:
- ✅ 핵심 도메인 보장: 모임 생성은 항상 성공 (채팅방 문제와 무관)
- ✅ 사용자 경험 개선: 모임 생성 즉시 응답, 채팅방은 비동기 생성
- ✅ 확장성: 다른 부가 기능도 이벤트 리스너로 추가 가능

**구현 파일**:
- `MeetupCreatedEvent.java`: 이벤트 클래스
- `MeetupChatRoomEventListener.java`: 이벤트 리스너
- `MeetupService.java`: 이벤트 발행 로직

**위치**: `ChatMessageService.markAsRead()` (line 149-190)

**문제점 분석**:

#### 1. 비효율적인 전체 메시지 조회

```java
// ⚠️ 문제: 전체 메시지를 조회하고 Java에서 필터링
List<ChatMessage> unreadMessages = chatMessageRepository

```
- **문제**: 채팅방의 모든 메시지를 조회 후 Java에서 필터링
- **영향**: 메시지가 많을수록 성능 저하 (수천~수만 건 조회)
- **현재 상태**: MessageReadStatus 기록 로직은 주석 처리되어 있음 (실제 사용 안 함)

#### 2. 프론트엔드에서 빈번한 호출
**위치**: `ChatRoom.js`
- **호출 시점**:
  - 메시지 조회 후 (line 56)
  - WebSocket으로 새 메시지 수신 시 (line 170)
  - 이미지 전송 후 (line 243, 248)

- **문제**: 읽음 처리가 매우 빈번하게 발생 → DB 부하 증가 가능

#### 3. 트랜잭션 범위가 넓음
- 참여자 조회 → 메시지 조회 → 필터링 → 반복 처리
- 불필요한 로직 포함 (MessageReadStatus는 사용 안 함)

**개선 방안**:

1. **불필요한 로직 제거**: MessageReadStatus 기록 로직 제거 (사용 안 함)
2. **읽음 처리 단순화**: `unreadCount = 0` 및 `lastReadMessage`만 업데이트
3. **프론트엔드 최적화**: 읽음 처리 호출 빈도 감소 (디바운싱 적용)
4. **배치 처리**: 여러 채팅방의 읽음 처리를 한 번에 처리

## 3. 👤 User (사용자) 도메인 (해결)

### ⚠️ 가장 심각한 문제: 닉네임 중복 체크 Race Condition

**위치**: `UsersService.createUser()`, `UsersService.updateMyUsername()`, `UsersService.setNickname()`

**문제 상황**:
```java
// UsersService.java:107-110
usersRepository.findByNickname(nickname)
    .ifPresent(existingUser -> {
        throw new RuntimeException("이미 사용 중인 닉네임입니다.");
    });
// ... 
Users saved = usersRepository.save(user); // ⚠️ 이 사이에 다른 요청이 같은 닉네임으로 가입 가능!
```

**시나리오**:
- 동시에 2명이 같은 닉네임으로 가입 시도
- 둘 다 중복 체크 통과
- 둘 다 가입 성공 → 닉네임 중복 발생!

**해결 방안**:

#### 3.1 DB 레벨 Unique 제약조건 (가장 확실)
```java
@Entity
@Table(name = "users", 
    uniqueConstraints = @UniqueConstraint(columnNames = {"nickname"}))
public class Users {
    @Column(length = 50, unique = true) // ✅ 이미 있음!
    private String nickname;
}
```

**현재 상태**: `Users.java:30`에 이미 `unique = true` 설정됨
- **문제**: 예외 처리 필요
```java
@Transactional
public UsersDTO createUser(UsersDTO dto) {
    try {
        // ... 중복 체크
        Users saved = usersRepository.save(user);
        return usersConverter.toDTO(saved);
    } catch (DataIntegrityViolationException e) {
        if (e.getMessage().contains("nickname")) {
            throw new RuntimeException("이미 사용 중인 닉네임입니다.");
        }
        throw e;
    }
}
```

#### 3.2 Pessimistic Lock 적용
```java
@Lock(LockModeType.PESSIMISTIC_WRITE)
Optional<Users> findByNicknameForUpdate(String nickname);
```

### 📊 인덱스 최적화

**필요한 인덱스**:
```sql
-- 사용자 조회 성능 향상 (이미 unique 제약조건으로 인덱스 생성됨)

-- CREATE UNIQUE INDEX idx_users_username ON users(username);

CREATE INDEX idx_users_location ON users(location); -- 위치 기반 검색
```

### ⚠️ 탈퇴한 사용자(Soft Delete)의 닉네임/username/email 재사용 불가 문제

**위치**: `UsersRepository.findByNickname()`, `findByUsername()`, `findByEmail()`  
**영향받는 메서드**: `createUser()`, `updateMyProfile()`, `updateMyUsername()`, `setNickname()`, `updateUser()`

**문제 상황**:
```java
// UsersRepository.java
Optional<Users> findByNickname(String nickname);  // ⚠️ isDeleted 필터링 없음!
Optional<Users> findByUsername(String username);  // ⚠️ isDeleted 필터링 없음!
Optional<Users> findByEmail(String email);        // ⚠️ isDeleted 필터링 없음!

// UsersService.java:107-110
usersRepository.findByNickname(nickname)
    .ifPresent(existingUser -> {
        throw new RuntimeException("이미 사용 중인 닉네임입니다.");
    });
// ⚠️ 탈퇴한 사용자도 조회됨!
```

**시나리오**:
1. 사용자 A가 "nickname1"로 가입 → 성공
2. 사용자 A가 탈퇴 (`isDeleted = true`)
3. 사용자 B가 "nickname1"로 가입 시도
4. `findByNickname("nickname1")`이 탈퇴한 사용자 A를 반환
5. 중복 체크에서 걸려서 가입 실패 ❌
6. **결과**: 탈퇴한 사용자의 닉네임을 다른 사용자가 재사용할 수 없음!

**영향**:
- 탈퇴한 사용자의 닉네임/username/email이 영구히 사용 불가능
- 사용자가 탈퇴 후 재가입 시 기존 닉네임 사용 불가
- 닉네임 고갈 가능성 (특히 짧은/인기 있는 닉네임)

**해결 방안**:

#### 3.3 Repository 메서드에 isDeleted 필터링 추가
```java
// UsersRepository.java
@Query("SELECT u FROM Users u WHERE u.nickname = :nickname AND (u.isDeleted = false OR u.isDeleted IS NULL)")
Optional<Users> findByNicknameAndNotDeleted(@Param("nickname") String nickname);

@Query("SELECT u FROM Users u WHERE u.username = :username AND (u.isDeleted = false OR u.isDeleted IS NULL)")
Optional<Users> findByUsernameAndNotDeleted(@Param("username") String username);

@Query("SELECT u FROM Users u WHERE u.email = :email AND (u.isDeleted = false OR u.isDeleted IS NULL)")
Optional<Users> findByEmailAndNotDeleted(@Param("email") String email);
```

또는 기존 메서드를 수정:
```java
@Query("SELECT u FROM Users u WHERE u.nickname = :nickname")
Optional<Users> findByNickname(@Param("nickname") String nickname);

// 서비스에서 필터링
public UsersDTO createUser(UsersDTO dto) {
    usersRepository.findByNickname(dto.getNickname())
        .filter(user -> !Boolean.TRUE.equals(user.getIsDeleted()))  // ✅ 탈퇴한 사용자 제외
        .ifPresent(existingUser -> {
            throw new RuntimeException("이미 사용 중인 닉네임입니다.");
        });
    // ...
}
```

**권장 해결책**: Repository 메서드에 명시적으로 `isDeleted = false` 조건 추가
- 코드 일관성 유지
- 실수로 탈퇴한 사용자를 조회하는 것을 방지
- 모든 중복 체크 로직에서 자동으로 적용

### 🔄 트랜잭션 개선

**현재**: `@Transactional` 클래스 레벨 적용 (line 28)
- **문제**: 읽기 전용 메서드도 쓰기 트랜잭션에 포함
- **개선**: 읽기 전용 메서드에 `@Transactional(readOnly = true)` 명시

## ✅ 해결 완료된 성능 최적화 항목

### 1. N+1 문제 해결: 배치 조회로 반응 정보 조회 최적화

**문제 상황**:
- 게시글 목록 조회 시 각 게시글마다 좋아요/싫어요 카운트를 개별 쿼리로 조회
- 1000개 게시글 기준: 1개 (게시글) + 2000개 (좋아요/싫어요) = **2001개 쿼리**
- 조회 시간: ~30초

**해결 방법**:
- 배치 조회로 IN 절을 활용한 집계 쿼리 사용
- 500개 단위로 배치 처리하여 IN 절 크기 제한

**구현 내용**:
```java
// BoardService.java
private List<BoardDTO> mapBoardsWithReactionsBatch(List<Board> boards) {
    // 1. 게시글 ID 목록 추출
    List<Long> boardIds = boards.stream()
        .map(Board::getIdx)
        .collect(Collectors.toList());
    
    // 2. 좋아요/싫어요 카운트 배치 조회 (IN 절)
    Map<Long, Map<ReactionType, Long>> reactionCountsMap = getReactionCountsBatch(boardIds);
    
    // 3. 게시글 DTO 변환 및 반응 정보 매핑
    return boards.stream()
        .map(board -> {
            BoardDTO dto = boardConverter.toDTO(board);
            Map<ReactionType, Long> counts = reactionCountsMap.getOrDefault(board.getIdx(), new HashMap<>());
            dto.setLikes(Math.toIntExact(counts.getOrDefault(LIKE, 0L)));
            dto.setDislikes(Math.toIntExact(counts.getOrDefault(DISLIKE, 0L)));
            return dto;
        })
        .collect(Collectors.toList());
}

private Map<Long, Map<ReactionType, Long>> getReactionCountsBatch(List<Long> boardIds) {
    final int BATCH_SIZE = 500;  // IN 절 크기 제한
    Map<Long, Map<ReactionType, Long>> countsMap = new HashMap<>();
    
    // IN 절을 500개 단위로 나누어 조회
    for (int i = 0; i < boardIds.size(); i += BATCH_SIZE) {
        int end = Math.min(i + BATCH_SIZE, boardIds.size());
        List<Long> batch = boardIds.subList(i, end);
        
        List<Object[]> results = boardReactionRepository.countByBoardsGroupByReactionType(batch);
        // 결과 파싱 및 Map 구성
    }
    
    return countsMap;
}
```

```java
               "MATCH(b.title, b.content) AGAINST(:kw IN BOOLEAN MODE) as relevance " +

               "WHERE b.is_deleted = false " +
```

## 🎯 우선순위별 개선 체크리스트

### 🔴 Critical (즉시 수정 필요)

1. **Meetup**: `currentParticipants` 증가/감소 Race Condition
   - Pessimistic Lock 또는 Optimistic Lock 적용
   - DB 제약조건 추가

2. **Report**: 중복 신고 체크 Race Condition
   - Unique 제약조건 추가
   - 예외 처리 개선

3. **User**: 닉네임 중복 체크 예외 처리
   - `DataIntegrityViolationException` 처리 추가

### 🟡 High (빠른 시일 내 수정)

4. **Chat**: 인덱스 추가
   - 메시지 조회 성능 향상

5. **Board**: 인덱스 추가
   - 검색 성능 향상

6. **Care**: 상태 변경 동시성 제어
   - Optimistic Lock 적용

### 🟢 Medium (점진적 개선)

7. **Location**: 공간 인덱스 적용
   - 위치 기반 검색 성능 향상

8. **Meetup**: 거리 계산 DB 레벨 처리
   - Java 필터링 → DB 쿼리로 변경

9. **Board**: 조회수 증가 비동기 처리
   - 성능 최적화
