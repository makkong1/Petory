# 펫코인 결제 알고리즘

## 1. 에스크로 상태 전이

**상태**: `HOLD` → `RELEASED` | `REFUNDED`

| 상태 | 의미 |
|------|------|
| HOLD | 거래 확정 후 ~ 서비스 완료 전, 코인 임시 보관 |
| RELEASED | 제공자에게 지급 완료 |
| REFUNDED | 요청자에게 환불 완료 |

**전이 규칙**:
- `HOLD` 상태에서만 `releaseToProvider` / `refundToRequester` 가능
- 이미 RELEASED/REFUNDED면 `PaymentConflictException` / `IllegalStateException`

---

## 2. 에스크로 생성 (createEscrow)

**입력**: CareRequest, CareApplication, requester, provider, amount

**흐름**:
1. amount 유효성 검사 (null, <= 0 → 예외)
2. 동일 CareRequest에 대한 기존 에스크로 존재 여부 확인 → 있으면 `escrowAlreadyExists`
3. `petCoinService.deductCoins(requester, amount, ...)` — 요청자 코인 차감
4. `PetCoinEscrow` 생성 (status=HOLD)
5. 저장

---

## 3. 지급 (releaseToProvider)

**흐름**:
1. `findByIdForUpdate(escrow.idx)` — 비관적 락
2. `status == HOLD` 검증
3. `petCoinService.payoutCoins(provider, amount, ...)` — 제공자에게 지급
4. `status = RELEASED`, `releasedAt = now`
5. 저장

---

## 4. 환불 (refundToRequester)

**흐름**:
1. `findByIdForUpdate(escrow.idx)` — 비관적 락
2. `status == HOLD` 검증
3. `petCoinService.refundCoins(requester, amount, ...)` — 요청자에게 환불
4. `status = REFUNDED`, `refundedAt = now`
5. 저장

---

## 5. 동시성 제어

- **비관적 락**: `findByIdForUpdate`, `findByCareRequestForUpdate` (PESSIMISTIC_WRITE)
- **목적**: 중복 지급/환불 방지, Race Condition 방지
- **트랜잭션**: 지급/환불 실패 시 CareRequest 상태 변경 롤백
