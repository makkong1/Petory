# M1 — 카피 분리 (응답성 개선)

- **일자**: 2026-05-13
- **소요**: M0 직후 (가이드 권장 ~1일, 실제로는 더 짧음)
- **목표**: 본 추천(`/recommend`)은 빠르게(< 1s) 받아 카드를 즉시 그리고, 시간이 걸리는 LLM 카피(`/recommend/copy`)는 별도 콜로 받아 코멘트 영역을 비동기로 채운다. LLM 이 다운돼도 카드는 정상.
- **가이드**: [`../PETORY-INTEGRATION.md`](../PETORY-INTEGRATION.md) §M1 체크리스트

## 1. 왜 (배경)

v3 의 `POST /recommend` 는 기본적으로 Ollama 를 호출하지 않아 < 1s 응답. 반면 `POST /recommend/copy` 는 Ollama 동기 대기로 30초 이상 걸릴 수 있다. 한 콜로 묶으면 추천 카드 전체가 30초 동안 안 보이는 문제가 생긴다. **두 콜로 쪼개서 시설/트렌드/사유 칩은 즉시, LLM 코멘트는 나중에** 보여주는 게 가이드의 핵심.

## 2. 무엇을 (변경 범위)

### 2.1 백엔드 — 신규 2개

```text

  RecommendCopyRequest.java   (신규)
  RecommendCopyResponse.java  (신규)
```

- `RecommendCopyRequest`: `{context, request_id, facilities[{name, distance_m}], trends[{keyword, score}], pet}`. `@Builder` + `@JsonInclude(NON_NULL)`. 프론트 ↔ 백엔드 / 백엔드 ↔ pet-data-api 사이에서 같은 record 재사용.
- `RecommendCopyResponse`: `{request_id, recommendation, source, generated_at}`. `source ∈ {"llm", "rule"}` — `rule` 이면 LLM 실패 시 폴백 카피.

### 2.2 백엔드 — 수정 4개

**`recommendation/dto/RecommendResponse.java`**

- `FacilityItem` 에 v3 신규 필드 5개 추가: `mentionCount`, `mentionScore`, `source`, `score`, `reasons`
- Jackson 기본 동작상 v2 응답에서는 null 로 채워짐 → 하위 호환 유지

**`recommendation/client/PetDataApiClient.java`**

- `RestClient` 를 2개로 분리: `recommendClient` (3s) / `copyClient` (35s)
- `buildClient()` 헬퍼 추출로 두 클라이언트 생성 로직 중복 제거
- `recommendCopy(RecommendCopyRequest)` 메서드 추가
- `X-Request-Id` 는 본 추천에서 만든 값을 카피 콜에서도 그대로 사용 (M3 트레이싱)

**`recommendation/service/RecommendService.java`**

- `recommendCopy(userId, body)` 추가
- 프론트에는 펫 정보를 넘기지 않고, 백엔드가 첫 호출과 동일한 로직 (`findPetInfo`) 으로 채워 일관성 보장
- `findPetInfo()` 헬퍼로 중복 제거

**`recommendation/controller/RecommendController.java`**

- `POST /api/recommend/copy` 엔드포인트 추가
- 인증 체크를 `requireUserId()` 헬퍼로 추출

### 2.3 프론트엔드 — 수정 2개

**`frontend/src/api/recommendApi.js`**

- `getCopy({requestId, context, facilities, trends})` 함수 추가
- payload 의 키는 백엔드 record 와 일치하도록 snake_case (`request_id`) 로 전송

**`frontend/src/components/Recommendation/RecommendCard.js`**

- 2단계 렌더링 도입
  1. 본 추천 응답 도착 시 facilities + trends + (있다면) 규칙 기반 카피 즉시 표시
  2. `/recommend/copy` 응답 도착 시 카피 영역을 LLM 카피로 덮어쓰기 (실패 시 폴백 카피 유지)
- `reasons` 라벨 매핑 함수와 칩 UI 추가
- `useEffect` cleanup 으로 `lat` / `lng` / `context` 가 빠르게 바뀔 때 stale state 방지

| 항목                                   | 결과                                                                     |
| -------------------------------------- | ------------------------------------------------------------------------ |
| `./gradlew compileJava` (M1 변경 반영) | ✅ exit 0                                                                |
| 백엔드 린터 오류                       | ✅ 없음                                                                  |

| 카피 콜 실패 시 카드 정상 렌더         | ✅ `catch` 에서 `setCopy(null)` → `data.recommendation` 폴백 그대로 사용 |

### 런타임 확인이 필요한 시나리오 (사용자가 실제 환경에서 검증)

1. `npm start` → 추천 페이지 진입 → 카드 헤더·시설 리스트·트렌드가 < 1s 안에 보이는가
2. 코멘트 영역에 "추천 코멘트 생성 중..." 인디케이터 → LLM 응답 도착 후 실제 코멘트로 교체되는가
3. pet-data-api 의 Ollama 컨테이너를 죽인 상태에서도 시설 카드는 정상이고, 카피는 규칙 기반 폴백 또는 hide 되는가
4. `reasons` 칩이 시설별로 한 줄에 적절히 표시되는가

## 5. 의도적으로 안 한 것

| 항목                                                                  | 이유                                                                                                       |
| --------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------- |
| `RecommendController` 의 `/copy` 경로에 별도 권한/Rate-limit          | 본 추천과 동일한 인증만 사용. 별도 정책 필요시 SecurityConfig 에서 추후 추가.                              |
| 카피 콜 결과를 백엔드에서 캐싱                                        | LLM 비용 절감 효과는 있지만, request_id 기반 캐시는 별도 설계 필요. 가이드 범위 밖.                        |
| `pet_breed_match` 라벨에 실제 품종 동적 삽입 (`"<품종> 케어"`)        | reason 문자열 자체에는 품종이 없어 별도 펫 컨텍스트 매핑이 필요. 가이드 표 그대로 따르면 정적 라벨로 충분. |
| `RecommendCopyRequest` 의 Controller-수신 DTO 와 client-송신 DTO 분리 | 모양이 동일하고 record 라 immutable. 분리 시 코드만 늘어남.                                                |
| 본 추천 결과 캐싱                                                     | 위치 기반이라 cache key 설계가 비효율. 추후 콜백(M3) 데이터가 쌓이면 다시 평가.                            |
