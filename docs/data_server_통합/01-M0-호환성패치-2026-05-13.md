# M0 — 호환성 패치

- **일자**: 2026-05-13
- **소요 시간**: ~30분 (가이드 권장과 동일)
- **목표**: pet-data-api v3 로 바뀌어도 Petory 추천 페이지가 깨지지 않게 하고, 본 추천 호출의 응답성을 개선한다.
- **가이드**: [`../PETORY-INTEGRATION.md`](../PETORY-INTEGRATION.md) §M0 체크리스트

## 1. 왜 (배경)

pet-data-api v3 에서 `POST /recommend` 의 기본 동작이 바뀌었다.

| 항목          | v2           | v3                                                               |
| ------------- | ------------ | ---------------------------------------------------------------- |
| LLM 카피 호출 | 항상         | `include_copy=true` 일 때만                                      |
| 응답 시간     | Ollama 30초+ | 기본 < 1초                                                       |
| 응답 필드     | 5개          | + `request_id`, `recommend_version`, (시설별) `score`, `reasons` |

즉시 깨지는 부분은 없으나, 본 추천 호출에 **여전히 30초+ 타임아웃** 을 두면 v3 의 빠른 응답을 활용하지 못한다. 또한 `request_id` 를 받아두지 않으면 M1(`/recommend/copy`)·M3(`/events/recommendation`) 에서 같은 호출을 추적할 수 없다.

## 2. 무엇을 (변경 범위)

**최소 변경 원칙(외과적 변경) 으로 두 파일만 수정.** v3 의 다른 신규 필드(`reasons`, `score`, `mention_*` 등)는 M1 에서 UI 와 함께 처리하므로 지금은 건드리지 않는다. `include_copy` 옵션도 현재 추천 카드가 카피 없이도 정상 렌더링되므로 추가하지 않는다.

### 2.1 `PetDataApiClient.java`

`backend/main/java/com/linkup/Petory/domain/recommendation/client/PetDataApiClient.java`

**1) 타임아웃 설정**

- 변경: `app.pet-data-api.timeout-ms` 설정 추가 (기본 3000), `SimpleClientHttpRequestFactory` 로 `connectTimeout` / `readTimeout` 적용
- 이유: v3 본 추천이 1초 미만으로 응답하는데 기존 30초+ 무한대기는 과함. `application.properties` 미수정으로도 default 3000ms 가 적용되도록 처리.

**2) 요청 ID 발급**

- 변경: 호출마다 16자리 UUID 로 `X-Request-Id` 헤더 생성·전송 + 로그에 동봉
- 이유: 응답의 `request_id` 와 매핑되어 M1 (`/recommend/copy`) · M3 (`/events/recommendation`) 에서 재사용 (가이드 §2.3).

**3) 로그 포맷 통일**

- 변경: 모든 로그에 `reqId={}` prefix 추가
- 이유: 추후 트레이싱 시 grep 용이.

### 2.2 `RecommendResponse.java`

`backend/main/java/com/linkup/Petory/domain/recommendation/dto/RecommendResponse.java`

- 신규 필드 2개 추가: `recommendVersion` (`@JsonProperty("recommend_version")`), `requestId` (`@JsonProperty("request_id")`).
- Jackson 기본 동작상 unknown 필드는 무시되므로 안 받아도 깨지진 않지만, 명시적으로 받아둬야 M1/M3 에서 사용 가능.
- `FacilityItem` 의 신규 필드(`reasons`, `score`, `mention_count` 등)는 M1 에서 UI 와 함께 추가.

## 5. 의도적으로 안 한 것

| 항목                                                      | 이유                                                                                                                          |
| --------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------- |
| `include_copy=true` 추가                                  | 현재 카드 UI 가 카피 없이도 렌더링됨. v3 기본(규칙 기반 카피)도 `recommendation` 필드를 채워줌. 카피 분리는 M1 의 책임.       |
| `FacilityItem` 의 `reasons`/`score`/`mention_*` 필드 추가 | UI 와 매핑되지 않은 상태에서 DTO 만 늘리면 미사용 필드 발생. M1 에서 칩 UI 와 함께.                                           |
| `application.properties` 직접 수정                        | gitignore 대상. `@Value` default 로 충분. 운영 환경에서 별도 조정이 필요하면 운영자가 `app.pet-data-api.timeout-ms` 추가.     |
| `RestClient` 빈 분리                                      | 현재 호출처 1군데(`RecommendService`). 추후 M1 에서 `/recommend/copy` 추가될 때 클라이언트가 두 개로 늘어나면 그때 분리 고민. |
