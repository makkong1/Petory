# A — 추천 카드 UI 강화 (pet-data-api 데이터 가시화)

- **일자**: 2026-05-13
- **상위 컨텍스트**: M1 에서 백엔드가 이미 받고 있는 v3 신규 필드들이 UI 까지 흐르지 못하는 문제 해결. 사용자 의견: "데이터서버에서 가공해 보낸 풍부한 신호가 정작 Petory 에는 안 보여 보여주기가 애매하다."
- **목표**: 백엔드 변경 0, 프론트 1개 파일만 수정. M1 에서 받기 시작한 `mention_count`, `source`, `score`, `reasons` 를 사용자 화면에서 의미있게 표현한다.

## 1. 왜 (배경)

M1 완료 시점 기준으로 pet-data-api v3 가 응답에 담아 보내는 신호 vs Petory UI 노출 현황:

| 응답 필드 | M1 직후 UI 노출 | 사용자가 받는 정보 |
|---|---|---|
| `trends[]` | ✅ "요즘 인기 키워드" 칩 | 시설과 분리되어 떠있음 |
| `facilities[].name`, `distance_m` | ✅ 표시 | 단순 텍스트 |
| `facilities[].score` | ✅ "★ 0.78" | 값의 의미 전달이 약함 |
| `facilities[].reasons` | ✅ 회색 칩 | 너무 작아서 눈에 안 띔. 트렌드 칩과 연결도 안 됨 |
| `facilities[].mention_count` | ❌ DTO 만 받고 미노출 | 블로그에서 N번 언급된 가게라는 신뢰 신호 누락 |
| `facilities[].source` | ❌ | 어느 데이터 출처인지 알 수 없음 |
| `recommendation` (LLM 카피) | ✅ 표시 | 본문 14px 텍스트 — 카드의 주인공인데 비중 ↓ |

→ **결론**: 데이터를 더 가져오기 전에, **이미 가지고 있는 데이터를 의미있게 보여주는** 작업이 먼저.

## 2. 무엇을 (변경 범위)

**`frontend/src/components/Recommendation/RecommendCard.js` 1 파일.**
백엔드·API·DTO 무수정.

### 2.1 LLM 카피 prominence ↑

`RecommendText` (단순 `<p>` 14px) → `RecommendQuote` (`<blockquote>` 스타일, 좌측 컬러 바 + 배경 + 15px). 카드의 시각적 중심으로 끌어올림.

### 2.2 시설 항목 레이아웃 재구성

기존 (한 줄 + 작은 점수 + 회색 칩):
```
해피독 미용실 · 320m              ★ 0.78
[가까워요] [스포팅컷 인기]
```

변경:
```
해피독 미용실                          추천도 78%
320m · [공공+카카오] · 블로그 언급 3건
[가까워요] [스포팅컷 인기]   ← trend_match 는 primary 컬러, 나머지는 중성
```

- **이름·점수 한 줄**: `score` 를 `★ 0.78` → `추천도 78%` 로 직관화. 우측 정렬, primary 컬러로 강조.
- **메타 한 줄**: 거리 + `source` 뱃지 + `mention_count` "블로그 언급 N건" 인라인.
- **`reasons` 칩**:
  - `trend_match:<키워드>` 라벨은 위쪽 트렌드 칩(`Tag`) 과 **같은 primary 색**으로 → "이 키워드가 인기라서 이 가게가 추천됐다" 가 시각적으로 즉시 전달.
  - 나머지(`distance`/`mention`/`history`/`pet_species_match`/`pet_breed_match`)는 중성 색.

### 2.3 `sourceLabel()` 매핑 함수

```js
function sourceLabel(source) {
  switch (source) {
    case 'public': return '공공';
    case 'kakao': return '카카오';
    case 'public+kakao': return '공공+카카오';
    default: return null;
  }
}
```

### 2.4 `isTrendReason()` 판별 헬퍼

`reasons` 배열에서 `trend_match:` 접두사를 가진 항목만 primary 컬러로 띄우기 위한 작은 헬퍼.

### 사용자 측 런타임 확인 시나리오

1. `npm start` → AI 추천 카드 진입
2. LLM 카피가 좌측 컬러 바와 배경색이 있는 인용구 스타일로 prominent 하게 표시되는가

4. `trend_match:스포팅컷` 칩이 상단 트렌드 키워드 칩과 같은 색으로 표시되어 둘 사이의 관계가 즉시 전달되는가

| 비-시설 카테고리(`snack`/`food`/`clothes`) 트렌드 전용 화면 | B 작업으로 분리. 트렌드 카테고리 페이지 신설은 별도 라우팅이라 카드 범위 밖. |

## 6. A 가 다음 단계에 끼치는 영향

이 A 작업으로 인해 후속 단계의 범위가 약간 조정됨:

| 후속 | A 전 가정 | A 후 조정 |
|---|---|---|
| B (트렌드 카테고리 페이지) | 트렌드 키워드를 시계열로 새 화면에 표시 | **추천 카드의 상단 트렌드 칩과 시각 일관성 유지 필요** — 같은 primary 컬러 토큰·라벨 포맷 사용. |
| D (M3 콜백) | 카드 클릭 시 `click` 이벤트 트래킹 | A 에서 시설 항목이 더 인터랙티브해 보이므로(`추천도 N%`, 칩 등) **시설 항목 자체에 `onClick` 부착**이 자연스러움. 클릭 핸들러가 들어갈 자리는 이미 마련됨. |
| 가이드 §M2 `/facilities/search` 도입 | 일반 시설 검색을 pet-data-api 로 마이그레이션 | A 결과로 추천 카드 자체가 풍부해졌으므로, **일반 검색은 Petory 자체 DB 유지**로 가닥. 사용자와 최종 컨펌 후 진행 또는 보류 결정. |

## 7. 다음 단계 (B) 미리보기

비-시설 카테고리 트렌드 페이지 신설:

- 백엔드: `GET /api/recommend/trends/{category}/timeseries` → pet-data-api `/trends/{category}/timeseries` 프록시 (`days=14&top_keywords=10` 기본).
- 프론트: 새 라우트 (`/trends/{category}`) 또는 기존 카테고리 페이지에 섹션 추가. 키워드 리스트 + (옵션) 라인 차트.
- 대상 카테고리: `snack`, `food`, `clothes` (공인 시설 없는 카테고리). `grooming`/`hospital` 도 보조 화면으로 확장 가능.

> 사용자 승인 받은 진행 방향대로 A 직후 B 시작.
