## 1. N+1 · 불필요 로딩

### 1-1. 목록 API에서 `participants` 접근

**현상:** 여러 목록 흐름이 `MeetupConverter#toDTO` / `#toDTOList`를 쓰고, 그 안에서 항상 `meetup.getParticipants()`를 호출한다.

**파일·코드:**

- `backend/main/java/com/linkup/Petory/domain/meetup/converter/MeetupConverter.java` — `toDTO` 내부 `getParticipants()` → 스트림 변환

**배경:** `Meetup.participants`는 `@OneToMany` LAZY + `@BatchSize(size = 50)` (`Meetup.java`). 목록 n건이면 batch로 추가 쿼리가 나가며, **UI에 participants가 필요 없어도** 로딩된다.

**목록 조회가 이 변환을 쓰는 서비스 예 (같은 컨버터):**

- `getAllMeetups` / `getAllMeetups(Pageable)` → `converter.toDTO` / `toDTOList`
- `getMeetupsByLocation`, `searchMeetupsByKeyword`, 주최자별 목록 등 — `MeetupService`에서 `converter.toDTOList` 호출부 참고

**상태:** **미반영.** 권장 방향은 문서 초안과 동일 — 목록 전용 DTO·`toSummaryDTO`(participants 미포함) 또는 “이미 participants가 초기화된 경우에만 매핑” 분기.

### 1-2. `findUpcomingMeetupsByUser`

**파일:** `SpringDataJpaMeetupParticipantsRepository.java`

**현행 JPQL (요약):** `JOIN FETCH mp.meetup m` + `JOIN FETCH m.organizer` + `JOIN FETCH mp.user` — **N+1 방지 적용됨.**

**호출:** `MeetupService` 등 **서비스 계층에서 참조 없음** (리포지토리·어댑터에만 존재). 사용 전 삭제 vs 유지는 팀 정책 문제.

## 2. 인덱스

### 2-1. DB 실측 vs 엔티티

- **실측 근거:** `docs/db_concept/db-concept-highlights-meetup.md` — `meetup` / `meetupparticipants` SHOW INDEX 요약.
- **엔티티 반영:** `Meetup.java` `@Table(indexes = { organizer_idx, idx_meetup_status, idx_meetup_date, idx_meetup_date_status, idx_meetup_location })`, `MeetupParticipants.java` (`user_idx`, `idx_meetupparticipants_user_liked_joined`).

**의미:** 새 환경에서 `ddl-auto=create`를 쓰더라도 **의도한 인덱스가 코드에 남고**, Flyway 없이 JPA만 쓰는 경우에도 재생성 시 참고 가능. 운영 DB는 기존대로 **별도 마이그레이션**으로 이미 맞춰진 상태라면 “코드 선언 = 문서화” 성격이 강함.

**남은 논의:** 다수 쿼리가 `(is_deleted = false OR is_deleted IS NULL)`을 함께 쓰는데, 인덱스 선행 컬럼에 `is_deleted`를 넣을지는 쿼리 플랜·카티널리티에 따라 별도 튜닝.

**키워드 검색:** `LIKE %keyword%`는 B-tree 인덱스로 해결 불가 — Full-Text 등 별도 전략 (프로젝트 규칙 문서와 동일).

## 4. 동시성

### 4-1. `joinMeetup` (반영 완료)

**파일:** `MeetupService.java` — `#joinMeetup`

1. **`meetupRepository.findByIdWithLock(meetupIdx)`** — `SpringDataJpaMeetupRepository#findByIdWithLock` (`@Lock(PESSIMISTIC_WRITE)`). 동일 모임 행에 대한 동시 참가 요청을 직렬화.
2. **`existsByMeetupIdxAndUserIdx`** — 락 이후 중복 여부 확인.
3. **`incrementParticipantsIfAvailable`** — 정원·상태 조건을 WHERE에 넣은 원자적 UPDATE.
4. **`save(participant)`** — `try/catch DataIntegrityViolationException`: 복합 PK 충돌 시 **`decrementParticipantsIfPositive`** 후 `alreadyJoined` (이중 클릭·극단 동시성에 대한 보험).

초안 문서의 “`findByIdWithOrganizer`만 쓰는 TOCTOU” 시나리오는 **현재 코드와 불일치** — 락 조회로 대체됨.

### 4-2. 정원 초과

`incrementParticipantsIfAvailable` / `decrementParticipantsIfPositive` — 타 사용자 동시 참여·취소에 대한 원자적 카운트 (`SpringDataJpaMeetupRepository`).

## 5. 페이징 · 대용량

### 5-1. `subList` 상한

**파일:** `MeetupService.java` — 상수 `MAX_LIST_SIZE = 500` (약 L60).  
`getMeetupsByLocation`, `searchMeetupsByKeyword`, 주최자별 목록 등에서 Repository가 **전부** 가져온 뒤 `meetups.subList(0, MAX_LIST_SIZE)` 적용 (예: 약 L470, L477, L508 근처 — 줄 번호는 변경될 수 있음).

**상태:** **미반영.** 개선은 `Pageable` / 네이티브·JPQL `LIMIT` 등으로 DB 단에서 자르기.

### 5-2. `getAllMeetups()` (인자 없음)

**파일:** `MeetupService.java` — `List<MeetupDTO> getAllMeetups()` (약 L223–227).

**컨트롤러:** `MeetupController`는 `getAllMeetups(Pageable)`만 호출하는 형태로 확인됨 — 비페이징 메서드는 **데드 코드에 가깝다.**

**권장:** `@Deprecated(forRemoval = true)` + JavaDoc으로 페이징 API 유도, 또는 삭제 후 영향 범위 검색.
