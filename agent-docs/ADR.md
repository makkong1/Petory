# Architecture Decision Records

### ADR-001: 멀티 모듈 대신 단일 모놀리식 Spring Boot 채택
**결정**: 초기 프로젝트 구조를 마이크로서비스나 멀티 모듈이 아닌 단일 모놀리식 애플리케이션으로 구성.
**이유**: 핵심 기능 구현과 빠른 MVP 검증에 집중하기 위함. 도메인 패키지 분리(Layered)로 결합도를 낮춰 추후 확장을 대비함.
**트레이드오프**: 대규모 트래픽 발생 시 특정 도메인(예: 채팅)만 독립적으로 스케일아웃하기는 어려우나, 초기 개발 속도와 인프라 관리 비용 측면에서 이득이 더 큼.

### ADR-002: 세션 대신 JWT 기반 인증/인가 채택
**결정**: 15분 만료 Access Token + DB 저장 Refresh Token (상태 비저장 형태) 채택.
**이유**: SPA(React) 프론트엔드와 REST API 백엔드 간 무상태(Stateless) 아키텍처를 유지하여 서버 확장 시 세션 불일치 문제를 차단하기 위함.
**트레이드오프**: 토큰 탈취 방어 로직 추가 및 Refresh Token DB 조회가 필요해 일부 오버헤드가 발생하지만, 독립적 클라이언트 확장을 보장함.

### ADR-003: Elasticsearch 대신 MySQL ST_Distance_Sphere 함수 활용
**결정**: 위치/거리 기반 복합 검색 시 별도 검색 엔진 대신 MySQL의 공간(Spatial) 함수 사용.
**이유**: 현 단계 데이터 규모 내에서는 RDBMS만으로 충분한 성능을 발휘하며, 외부 시스템 도입 시 발생하는 관리/학습 곡선을 피하기 위함.
**트레이드오프**: 수백만 건 단위의 초고속 공간 검색에서는 불리할 수 있으나, 현재 운영/인프라 리소스를 극도로 절약할 수 있음.

### ADR-004: 실시간 통신을 위한 STOMP 웹소켓 채택
**결정**: 순수 WebSocket이 아닌 Spring WebSocket STOMP 프로토콜 사용.
**이유**: Pub/Sub (채팅방/메시징) 라우팅을 브로커에 위임해 복잡한 방 관리 세션 로직을 서버에서 직접 짜지 않기 위함.
**트레이드오프**: STOMP 프로토콜 오버헤드가 다소 추가되지만, 규격화된 메시지 형식과 유지보수성이 크게 향상됨.
